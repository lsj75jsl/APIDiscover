APIDiscover 스캔 틱 동작 매뉴얼

운영자(엔지니어)용 — 주기 스캔이 도메인을 조금씩 나눠 처리하는 방식과 전체 판정 파이프라인(선발→제외→그룹치환→샘플링 윈도우→delta-skip→배칭 조회→분석)의 직관적 이해.

틱(tick) 처리 모델 롤링 샘플링(D66) 배칭·엣지그룹(D63/D65) domains-per-tick off-peak 상향 관련 코드 · 설정 변경법
이 문서의 범위 주기 스캔의 "한 번에 다 훑지 않고 조금씩 나눠 처리"하는 틱 모델과, 한 틱에 몇 개를 처리할지 정하는 domains-per-tick, 그리고 D58~D66 으로 확장된 판정 파이프라인 전체(활성 우선 선발·엣지 제외/그룹치환·롤링 샘플링·delta-skip·배칭 조회)를 설명한다. 부하 전체 정책(A watermark · E 전역 레이트 가드 등)은 수집 운영 매뉴얼을 참조. 본 문서 수치는 실제 배포값(adc.yaml env override) 기준이며, 기본값(application.yml)과 구분해 표기한다.

  1. 큰 그림 — 왜 "틱" 방식인가

  3. domains-per-tick = 평상시 한 틱의 몫
  4. off-peak-domains-per-tick = 밤에만 몫을 키우는 다이얼
  5. 하루 타임라인으로 보면
  6. 실제로는 이론치보다 적게 돕니다
  7. 첫 스캔은 얼마나 과거까지 보는가 — initial-backfill
  8. ★실제 스캔 한 사이클 — 시계열로 따라가기 (초보자용)
  9. 관련 코드 정보 (file:line)
  10. 설정값 변경 방법

1. 큰 그림 — 왜 "틱" 방식인가

도메인이 53,718개입니다. 이걸 한 번에 다 스캔하면 Loki 에 순간적으로 수만 건 쿼리가 몰려 운영 Loki 가 죽습니다. 그래서 "한 번에 다 훑기"를 버리고, 짧은 주기마다 조금씩 나눠 처리하는 방식을 씁니다. 이 "조금씩"의 주기가 tick, 한 틱에서 처리할 도메인 개수가 domains-per-tick 입니다.

은행 창구 비유 대기번호표를 뽑은 손님(도메인)이 53,718명 있고, 창구는 약 1분마다 한 번(tick) 열려서 번호 순으로 앞에서 500명(domains-per-tick) 만 부릅니다. 나머지는 다음 회차로 이월됩니다.

2. 한 틱이 실제로 하는 일 DiscoveryScheduler.scanTick

약 1분마다(배포값 PT1M — fixedDelay 라 이전 회차 완료 후 1분) 이 루프가 돕니다.

1. 지금이 off-peak 시간대인가? 판정   (밤 01~06시 Asia/Seoul 인가)
2. K 결정 = 500   ← domains-per-tick (현 배포는 주간·off-peak 동일 500)
3. K개 선발 — ★신규 방문 확정 사이트 우선 + 나머지(통과 예정) 최소 1/5 예약  (ScanSelector, D64)
4. 꺼낸 K개를 하나씩:
     - Loki 쿼리 예산 남았나? 없으면 즉시 중단, 나머지는 다음 틱으로  (LokiBudget)
     - 이 도메인의 "다음 차례 시각"을 미래로 밀어둠 (재선택 방지·기아 방지)
     - 실제 Loki 조회 + API 평가 실행

핵심 두 가지입니다.

  • "번호 순" = nextScanDueAt asc nulls first 정렬. 가장 오래 스캔 안 된(=가장 밀린) 도메인이 맨 앞. 아직 한 번도 안 본 도메인(null)이 최우선. 그래서 먼저 밀린 게 먼저 처리(FIFO), 굶는 도메인 없음.
  • 한 틱에 딱 K개 = PageRequest.of(0, k) (ScanSelector.java:44-45). 큐에 5만 개가 밀려 있어도 이번 틱엔 앞에서 K개만 가져옵니다.

3. domains-per-tick = 평상시 한 틱의 몫

  • 현재 500(배포값). tick 1분(fixedDelay·처리시간 포함 실제 ~2분 간격) → 이론상 시간당 ~15,000 도메인 을 검토하는 속도. 검토 대부분은 로그를 안 읽는 값싼 통과(§8 delta-skip)라 가능한 수치입니다.
  • 이 값이 처리 속도의 메인 다이얼입니다. 올리면 순회가 빨라지지만 Loki 쿼리도 비례해 늘어 부하↑. 내리면 부하↓·순회 느려짐.

4. off-peak-domains-per-tick = 밤에만 몫을 키우는 다이얼

off-peak = 트래픽이 적은 새벽 시간대(현재 01:00–06:00, Asia/Seoul)를 말합니다. 이 시간엔 운영 Loki 자체도 한가하므로, 밀린 도메인을 더 빨리 따라잡으라고(backfill) 한 틱의 몫을 키웁니다.

  • 평시(주간): 틱당 500 (domains-per-tick).
  • off-peak(새벽 01~06시): 틱당 500 (off-peak-domains-per-tick) — ★현 배포는 주야 동일로 설정(야간 몰아치기 방지, 사용자 확정). 값만 바꾸면 야간 가속을 다시 켤 수 있는 기능은 그대로입니다.

코드로는 딱 한 줄 삼항입니다 (ScanSelector.java:44).

int k = offPeak ? props.scan().offPeakDomainsPerTick()   // 새벽 (현 배포 500)
                : props.scan().domainsPerTick();          // 평시 (현 배포 500)

즉 "누구를 뽑느냐(밀린 순)"는 밤낮 동일하고, "몇 개 뽑느냐(K)"만 밤에 커집니다. 부하 여유가 있는 시간에 몰아서 백로그를 태우는 전략입니다. (참고. 조회 윈도우 off-peak-max-window 도 현 배포는 주간과 동일 PT30M 입니다 — 과거 PT24H 대량조회가 Loki 과부하를 유발해 축소. DiscoveryScheduler.java:52.)

5. 하루 타임라인으로 보면

★현 배포는 주야 동일(틱당 500·윈도우 PT30M)입니다 — 야간 몰아치기 방지를 위해 off-peak 가속을 평시와 같게 맞췄습니다(사용자 확정). 아래는 그 상태의 하루입니다.

gantt title 하루 스캔 처리량 (현 배포 — 주야 동일) dateFormat HH:mm axisFormat %H:%M section 주야 동일 500개/틱 · 윈도우 30분 00–24시 :done, p1, 00:00, 24h section (참고) off-peak 01–06시 — 별도 가속 없음 01–06시 :o1, 01:00, 5h

off-peak 가속(틱당 K·윈도우 상향)은 기능으로 남아 있습니다 — env 값만 바꾸면 야간에만 더 처리하게 되돌릴 수 있습니다(§10).

6. 실제로는 이론치보다 적게 돕니다

몇 개의 상한이 더 있어서, 이론치가 그대로 안 나옵니다.

  • max-queries-per-hour(현 배포 6000) — 틱 도중 이 쿼리 예산이 바닥나면 budget.hasBudget()==false 로 그 틱을 조기 종료하고 나머지는 이월(DiscoveryScheduler.java:55-58). Loki 타임아웃/연결실패도 예산에 계상되고 적응형 감속이 걸립니다(D58).
  • 이미 최근에 본 도메인은 nextScanDueAt 가 미래라 큐 뒤로 가서 이번 틱 대상에서 빠짐.

대신 요즘은 delta-driven skip(§8) 덕분에 검토 대부분이 Loki 조회 없는 값싼 통과라, 검토 속도는 높게 유지하면서 실제 Loki 조회만 예산 안에서 이뤄집니다.

정리 domains-per-tick = 평상시 3분마다 처리할 도메인 수, off-peak-domains-per-tick = 새벽 저부하 시간대에 대신 쓰는 더 큰 수입니다. 둘 다 "속도 다이얼"이고, 뽑는 순서(가장 밀린 순)는 바꾸지 않습니다.

7. 첫 스캔은 얼마나 과거까지 보는가 — initial-backfill

여기서 "언제부터 언제까지" 시간 구간을 정하는 규칙이 하나 더 있습니다. 자동 주기 스캔은 도메인을 처음 스캔할 때 now − initial-backfill(기본 7일) 지점부터 시작합니다. 로그가 아무리 오래 쌓여 있어도, 그보다 과거는 아예 읽지 않습니다.

도메인마다 "어디까지 읽었나"를 기록하는 커서(watermark)가 있습니다. 시간 구간은 이렇게 정해집니다 (DiscoveryJobService.windowFor:484-498).

  • 이후 워터마크가 max-window(평시 30분/새벽 24h)씩 현재 쪽으로 전진하며 최신을 따라잡습니다.
initial-backfill 은 "얼마나 과거까지 볼지"의 유일한 손잡이
  • 기본 P7D → 첫 스캔은 7일 과거까지만.
  • 더 깊게 보려면 첫 스캔 전에 initial-backfill 을 키워야 합니다(예 P30D).
  • ★소급 적용 안 됨 — 이미 스캔된 도메인은 워터마크가 잡혀 있어 이 값을 바꿔도 과거로 되돌아가지 않습니다.
  • 온디맨드(수동) 스캔으로 과거를 보려 해도 그 윈도우는 max-window 상한에 걸려 멀리 못 갑니다(onDemandWindow:181-189).
  • 운영 Loki 자체 보존기간도 보통 10년이 아니므로, 시작점을 과거로 당겨도 Loki 에 남아있는 기간까지만 조회됩니다.
★샘플링 모드(D66, 현 배포)에서는 이 장이 주기 스캔에 적용되지 않습니다 현 배포는 롤링 샘플링이 켜져 있어(§8.7) 주기 스캔이 항상 "최신 10분"만 읽습니다 — 신규 도메인도 과거 7일 백필 없이 최신 구간부터 시작합니다(1,008청크 백필 방지). initial-backfill 은 샘플링을 끈(gap-free) 구성에서만 유효합니다.

워터마크 증분·per-scan 윈도우 상한(A)의 전체 정책은 수집 운영 매뉴얼 §3.3 참조.

8. ★실제 스캔 한 사이클 — 시계열로 따라가기 (초보자용)

이 장은 프로그램을 몰라도 이해할 수 있도록, 시계를 14:00에 맞춰 놓고 5분 동안 시스템 안에서 실제로 무슨 일이 벌어지는지 순서대로 따라갑니다. 모든 그림은 마우스를 올리면 나오는 🔍 확대 버튼(또는 그림 클릭)으로 크게 볼 수 있습니다.

8.6 왜 이렇게 도는가 — 설계 이유 세 가지

  • 도서관(운영 Loki) 보호 — 조회는 "방문이 있었던 사이트"에만, 그것도 시간당 상한 안에서만.
  • 기록 무손실 — 책갈피(워터마크)가 "여기까지 읽음"을 보장해, 밀려도 이어 읽지 빠뜨리지 않습니다.
  • 신선도 — 방문 많은 사이트는 ~30분마다 다시 보고, 방문 없는 사이트는 공짜로 통과시켜 전체가 실시간에 가깝게 유지됩니다.

8.7 롤링 샘플링 (D66) — 왜 전부 읽지 않는가

활성 사이트 1.1만 개의 로그 전부를 빠짐없이 읽으려면 시간당 ~2.6만 윈도우가 필요한데, 운영 도서관(Loki) 보호 예산으로는 ~7천이 한계입니다 — 전부-읽기를 고집하면 영원히 뒤처집니다(실측: 평균 17시간 지연·계속 발산). 그래서 표본 조사로 전환했습니다.

  • 방식 — 사이트를 방문할 때마다 "그 시점의 최신 10분"만 읽고, 책갈피를 지금−10분으로 점프. 활성 사이트는 30분마다 재방문 → 시간당 2회 × 10분 = 20분치(33%) 표본.
  • 얻는 것 — 발산 정지. 활성 사이트의 결과가 항상 30분 이내 최신. 자주 호출되는 API 는 어느 표본에든 잡힘.
  • 포기하는 것(정직) — 표본 밖 40분에만 발생한 호출은 그 시간엔 못 봄. 드문 엔드포인트는 발견이 늦어질 수 있음(사이트마다 표본 시점이 자연히 흩어져 결정적 사각지대는 없음 — 반복 호출은 결국 잡힘). 호출수(hits) 지표는 표본 기반(~1/3 관측).
  • 고정 시각표가 아닌 이유 — "매시 0~10분/30~40분" 고정이면 :15 에 도는 정기 작업을 영원히 놓침(결정적 사각지대). 롤링(도메인별 방문 시점 기준)은 이 문제가 없고, 도서관이 느려져도 일정이 붕괴하지 않음.

10. 설정값 변경 방법

핵심 — 런타임 REST API 없음, 소스 하드코딩도 아님 이 노브들(domains-per-tick·off-peak-domains-per-tick·tick-interval·max-queries-per-hour 등)은 Spring 외부 설정이다. 값 자체는 application.yml 에 있고 adc.yaml env 로 덮어쓴다. 재빌드는 불요지만, 살아있는 상태에서 API 로 바꿀 수는 없고 pod 재시작(podman play kube)으로만 반영된다. (예: LokiBudget 은 생성자에서 값을 1회만 읽는다 — LokiBudget.java:33-34.)

10.2 방법 A — 배포 VM 에서 env 만 바꿔 재적용 (권장 · 재빌드 불요)

이미지는 그대로 두고 adc.yaml 의 env 값만 고쳐 pod 를 재시작한다. 예로 평시 처리량을 500→200 으로 낮추려면.

# 1) 로컬 adc.yaml 편집 — 해당 env value 수정
#    - name: APIDISCOVER_SCAN_DOMAINSPERTICK
#      value: "200"          # 500 → 200 (예시)

# 2) VM 으로 복사
scp adc.yaml root@192.168.8.197:/root/adc.yaml

# 3) pod 재시작 (이미지 불변, 설정만 재적용)
ssh root@192.168.8.197 '
  podman pod stop adc &&
  podman pod rm adc &&
  podman play kube /root/adc.yaml
'

# 4) 기동·반영 확인
ssh root@192.168.8.197 '
  curl -s http://localhost:8080/actuator/health;
  curl -s http://localhost:8080/actuator/configprops | grep -o "\"domainsPerTick\":[0-9]*"
'
확인 팁 현재 적용된 값은 configprops 액추에이터로 확인한다(내부 노출: adc.yaml 의 MANAGEMENT_ENDPOINTS_WEB_EXPOSURE_INCLUDE 에 configprops 포함). max-queries-per-hour 실소비는 /actuator/metrics/loki.queries(누적 카운터)로 관찰.

10.3 방법 B — 기본값 자체를 바꾸기 (재빌드 필요)

모든 환경의 기본값을 바꾸려면 src/main/resources/application.yml 의 scan: 블록을 수정하고 이미지를 재빌드한다. 배포에 env override 가 있으면 env 가 우선하므로 배포값도 함께 확인할 것.

# src/main/resources/application.yml
scan:
  domains-per-tick: 100          # 기본값 수정
  off-peak-domains-per-tick: 500
  tick-interval: "PT5M"
  max-queries-per-hour: 3000

# 재빌드
JAVA_HOME=/usr/lib/jvm/java-21-openjdk ./gradlew build

10.4 값 조정 지침 — 부하 vs 순회 속도

목표조정효과
순회를 빠르게(24h 내 전체)domains-per-tick ↑ (또는 tick-interval ↓)시간당 스캔 도메인↑. 병목은 쿼리캡이 아니라 이 값이므로 순회 속도의 주 레버. Loki 부하도 비례 증가.
Loki 부하 낮추기domains-per-tick ↓시간당 쿼리↓. max-queries-per-hour 캡보다 더 직접적인 레버(캡은 천장일 뿐, 실소비는 이 값이 만든다).
부하 스파이크만 상한max-queries-per-hour 를 실소비 약간 위로현 배포는 기본 3000 — 상향은 Loki 부하 직접 증가라 신중(과부하 이력). 쿼리 로그(elapsedMs)·loki.errors 보며 단계적으로만.
새벽에만 몰아 처리off-peak-domains-per-tick ↑저부하 시간대 백로그 소진 가속. 주간 부하는 불변.
과거를 더 깊게 스캔schedule.initial-backfill ↑ (첫 스캔 전에)첫 스캔 시작점을 과거로 당김. ★샘플링 모드(현 배포)에선 주기 스캔에 미적용(§7 주의 박스).
표본 커버리지 조정sample-window ↑/↓ · active-interval ↓/↑커버리지 = sample-window ÷ 재방문주기(현 10m/30m=33%). 올리면 커버리지↑·쿼리 비용↑(예산과 함께 조정).
