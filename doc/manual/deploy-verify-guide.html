APIDiscover 테스트서버 배포 검증 가이드

app + postgres 2컨테이너 1 pod(podman, hostNetwork)로 테스트 VM 에 배포하고, Loki 도메인 수집·CLI CSV 를 검증하는 절차. 권위 문서 doc/32-container-deploy-runbook.md·DECISIONS D44. 대상 Loki(192.168.8.100:3200)는 운영 서버 — 대용량 확인은 off-peak(01:00–06:00), 창/limit 작게.

0. 사전 준비 (테스트 VM, rootful=root 로그인)

mkdir -p /opt/adc /opt/adc-exports /opt/adc-log
chcon -Rt container_file_t /opt/adc /opt/adc-exports /opt/adc-log   # SELinux Enforcing(Rocky·el8 등) 시 컨테이너 접근 허용
# /opt/adc 는 비워둔다(postgres entrypoint 가 root 로 PGDATA 하위 pgdata 를 chown·initdb).
# /opt/adc-log 는 adc.yaml 의 app volumeMount(D58, 파일 로그 외부 영속) — 사전 생성+relabel 필수.

VM 요건: podman 설치, host 가 LAN Loki(192.168.8.100:3200) 도달 가능, 5432/8080 포트 미사용.

1. 이미지 빌드·전송 (★개별 tar)

podman build -t localhost/apidiscover:test .          # 멀티스테이지 bootJar→JRE + wait-for-db.sh

# ★이미지는 반드시 개별 tar 로 저장·전송한다
podman save localhost/apidiscover:test -o app.tar
podman save docker.io/library/postgres:16-alpine -o pg.tar
scp app.tar pg.tar <vm>:/root/
ssh <vm> 'podman load -i /root/app.tar; podman load -i /root/pg.tar'
⚠️ 결합 저장 금지. podman save img1 img2 -o one.tar 는 load 시 두 태그가 한 이미지로 뭉개져(태그 collision) postgres 태그가 app 이미지를 가리키는 사고가 난다 → app 이 빈 DB 에 접속해 크래시 루프(재시작 폭주). 로드 후 podman images 에서 app(약 401MB)·postgres(약 297MB)의 이미지 ID 가 서로 다른지 반드시 확인. (VM 인터넷 가용 시 postgres 는 podman pull 도 가능.)

2. 기동

podman play kube adc.yaml
podman pod ps && podman ps --pod        # adc pod 의 db·app Running
  • hostNetwork: true — postgres=host:5432, app=host:8080 직접 바인드. app↔db 는 동일 host netns localhost:5432.
  • app 은 wait-for-db.sh 로 localhost:5432 준비를 최대 120s 대기 후 기동 → DB 미준비로 인한 크래시 루프 없음.
  • container 프로파일 → PostgreSQL·ddl-auto=update. 첫 기동 시 postgres 가 /opt/adc/pgdata 에 initdb.

4. CLI CSV 내보내기 (one-off, 서빙 무부하)

podman run --rm --network host -v /opt/adc-exports:/exports \
  localhost/apidiscover:test -domain -export <domain>
# → /opt/adc-exports/<domain>-<epochmillis>.csv
  • 인자가 ENTRYPOINT(wait-for-db.sh→java -jar /app/app.jar) 뒤에 append → CLI 모드(웹·스케줄러 미기동, Loki 미호출, 1 명령 후 종료).
  • exit: 0 성공 / 2 도메인 미지정 / 3 도메인 미존재·검출 0건 / 4 쓰기 실패.
  • CSV 15열+헤더: host·method·path_template·status·source·confidence·severity·estimated·spec_ref·preflight_ambiguous·low_confidence·param_query·param_path·first_seen·last_seen. (score 는 범위 밖.)
CSV 에 내용이 나오려면 도메인별 엔드포인트 스캔(별도 스케줄, 기본 PT1H)이 discovered_endpoint 를 채운 뒤여야 한다. 갓 디스커버리된 도메인은 다음 스캔 사이클 전까지 "검출 0건"(exit 3)이 정상.

7. 한계 / 주의

  • max-domains-per-run — 1회 run 등록 도메인 캡. 현재 0(무제한·전수 등록)이라 롱테일 드롭 없음(과거 기본 200 → 0 으로 상향). 대상 축소는 대신 excluded-hostnames(AAJ·P* 계열 엣지 제외, D62/D69)로 한다. 캡은 DB 업서트량만 제한(Loki 부하 아님 — 벡터는 이미 수신).
  • hostNetwork — postgres 5432(adc/adc)가 VM 인터페이스에 노출된다. 내부 테스트 VM 범위에선 수용 가능하나, VM 공유/노출 시 host 방화벽·listen_addresses 하드닝을 고려.
  • 운영 Loki — 스캔·디스커버리는 운영 Loki 를 호출한다. LokiClient 의 부하보호(윈도우·page-limit·throttle·동시성·백오프)를 우회하지 말 것. 대용량 확인은 off-peak.

8. DB 성능 유지보수 (D75~D77)

항목내용fresh DB 조치
domain_hostnames.host 인덱스 (D75)@ElementCollection(EAGER) 라 도메인 로드마다 where host=? 조회 — 인덱스 없으면 전체 seq scan 으로 PG 백엔드 CPU 포화(FK 는 자식 인덱스 자동생성 안 함)소스 @Index 로 ddl-auto 자동 생성 — 조치 불요
discovered_endpoint autovacuum (D76)수백만 행·고빈도 upsert — 전역 임계(50+0.2×행)에서야 발화 → 대형 vacuum CPU 스파이크DB-레벨(소스 표현 불가) — fresh DB 에 1회 재적용: ALTER TABLE discovered_endpoint SET (autovacuum_vacuum_scale_factor=0.05, autovacuum_vacuum_threshold=10000, autovacuum_analyze_scale_factor=0.05, autovacuum_analyze_threshold=10000);
bloat 정리 (D76/D77)never-vacuum 시 dead tuple·인덱스 bloat 누적필요 시 VACUUM (ANALYZE, PARALLEL 0) discovered_endpoint; · 인덱스 SET max_parallel_maintenance_workers=0; REINDEX INDEX CONCURRENTLY <idx>;
⚠️ 병렬 유지보수와 /dev/shm. 컨테이너 기본 /dev/shm(64MB)은 병렬 VACUUM/REINDEX 의 동적 공유메모리에 부족해 실패한다(could not resize shared memory … No space left on device). 비병렬로 회피: VACUUM (PARALLEL 0) / SET max_parallel_maintenance_workers=0 후 REINDEX. autovacuum 은 병렬을 안 써서 무관. emptyDir(medium:Memory) 로 /dev/shm 확대는 el8 SELinux 가 tmpfs 를 relabel 하지 않아 PG 크래시루프 → 사용 금지(D77).
