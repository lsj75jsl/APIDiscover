2. ① 도메인 수집 (디스커버리) 구현됨

근거: doc/30 · DECISIONS D42 / D44. 담당 컴포넌트: DomainDiscoveryScheduler · DomainDiscoveryService · DomainUpserter · LokiClient(instant 쿼리).

수집 중 access log 에서 API 도메인(Host 헤더) 을 자동 열거해 DomainConfig 를 채운다. 수동 등록 부담을 없애는 것이 목적이다. 핵심은 원시 로그를 받지 않고 Loki 서버측에서 집계한 (도메인 × 엣지 hostname) 카운트만 수신한다는 점 — 그래서 매우 가볍다.

2.1 동작 방식 — 서버측 집계 쿼리

도메인은 Loki 라벨이 아니라 로그 라인 내용(필드 15 host / 16 real_host, 구분자 ^|^)에 있다. 따라서 서버에서 라인을 pattern 으로 파싱·집계하되, 도메인 합성(coalesce)·필터·검증은 클라이언트(Java) 에서 한다.

sum by (host, real_host, hostname) (
  count_over_time(
    {job="access_log"}
      | pattern "<_>^|^<_>^|^ ... (인덱스 0–14 skip) ... ^|^<host>^|^<real_host>^|^<_>"
    [W]
  )
)
# instant 벡터 쿼리(/loki/api/v1/query) — 응답은 집계 1벡터(라인 미전송)
★ coalesce 는 반드시 클라이언트에서 (실측 교훈, D44) 서버측 | label_format domain=coalesce(...) 를 붙이면 Go 템플릿이 라인마다 평가돼 약 10배 느려진다(실 Loki 5분창 2.2s → 20.2s) → 부트스트랩 1h·롤링 12m 모두 query-timeout(30s) 초과 → 디스커버리 실패. 그래서 서버는 host·real_host·hostname 으로만 group by 하고, 클라이언트가 firstNonEmpty(normalizeDomain(host), normalizeDomain(real_host)) 로 합성한다(LogLineParser 와 동일 의미: host 빈/- → real_host 폴백).

같은 도메인이 여러 (host, real_host)·여러 hostname 으로 나오면 클라이언트가 coalesce 후 도메인 키로 합산(hostnames 합집합·count 합)해 DomainConfig.hostnames(엣지 서버 집합)를 자동으로 채운다. 이 hostnames 가 ②스캔에서 어느 엣지를 조회할지 결정한다.

2.3 주기 · 설정 키

스캔과 별도 스케줄러(DomainDiscoveryScheduler)로 돌며, initial-delay 로 스캔과 시작 시점을 어긋나게(stagger) 해 동시 Loki 타격을 피한다. 첫 실행(빈 도메인 DB)만 bootstrap-window(1h) 로 초기 도메인을 일괄 확보하고 이후 롤링한다.

3. ② 도메인별 API 정보 수집 (스캔) 구현됨

근거: doc/33 · DECISIONS D45(B+A+E) · D48(C+D+F 티어링) · D56~D69(운영 튜닝). 담당: DiscoveryScheduler.scanTick() · ScanSelector.selectForTick() · ScanTier.effectiveInterval · OffPeakWindow · DiscoveryJobService.collectBounded/runScanBatched · LokiBudget · LokiClient.

3.1 왜 부하 정책이 필요한가

디스커버리가 ~13,920+ 도메인을 무제한 등록하므로, 과거처럼 @Scheduled(PT1H) 로 enabled 전 도메인을 전수 순차 스캔하면 운영 Loki 가 지속 과부하된다. 특히 gap-free 백필에서 미스캔 도메인의 첫 윈도우가 end − initial-backfill(P7D) = 7일치가 되면, 7일/슬라이스 만큼의 쿼리 × hostnames 를 당긴다(14k 동시 백필 = 재앙).

그래서 현재 스캐너는 전수 순회를 due 기반 슬라이스 순회로 바꾸고, 여러 축으로 부하를 통제한다 — B(틱당 예산·티어링), A(윈도우 상한·롤링 샘플링·delta-skip), E(전역 하드캡·per-scan 하드캡) + 운영 튜닝(무접속 중단·엣지 배칭·엣지 그룹). 기본 설정에서 대부분의 도메인은 매 틱 delta-skip 로 0쿼리이며, 실조회만 배칭돼 Loki 청크 재읽기를 1/N 로 줄인다.

커서(lastScanAttemptAt·nextScanDueAt)는 attempt 마다 먼저 전진하고, 이어 스캔한다(재선택·기아 방지). 예산 소진 또는 K 도달 시 틱 종료 → 나머지는 다음 틱이 이어받음(이월).

3.2 B — 틱당 예산 + due 기반 선택(티어링 + 활성 우선)

한 틱에 전 도메인을 금지하고, 짧은 틱마다 예산(K) 내에서 "스캔 due 가 도래한 도메인부터" 처리한다.

  • 선택: ScanSelector.selectForTick() = enabled + due 도래(nextScanDueAt null=즉시 또는 ≤ now)를 nextScanDueAt 오름차순(nulls first = 가장 밀린 순) 상위 domains-per-tick(K) 개. due 값은 티어(C)·dormant(F)·override 가 collapse 한 단일 타임스탬프다(3.5, ScanTier.effectiveInterval).
  • 활성 우선(D64): 후보 중 워터마크 이후 신규 트래픽이 확정된(=실제 Loki 조회가 필요한) 도메인을 먼저 채우고, 신규 트래픽 없는(delta-skip 대상) 도메인은 틱당 최소 K/5 만 예약한다 → 활성 도메인의 스캔 지연 최소화, 기아 방지.
  • 커서 = 영속 타임스탬프: lastScanAttemptAt(관측)·nextScanDueAt(due, @Index) — 둘 다 DB 영속(재기동 생존). 메모리 인덱스 미채택.
  • ★커서는 attempt 마다 전진(skip·실패 포함): touchScanSchedule(host, now, now+effectiveInterval) 로 lastScanAttemptAt=now + nextScanDueAt 갱신. up-to-date/skip 도메인이 매 틱 재선택되는 예산 낭비를 막고 FIFO 기아를 방지.
  • 무접속 중단(D59, inactive-after=P3D): 마지막 실 access log 관측(lastSeenAt)이 inactive-after 보다 오래된 도메인은 선택에서 제외(수집·평가 안 함). 경량 디스커버리는 계속 돌므로 트래픽 재개 시 lastSeenAt 이 갱신되면 자동 복귀(수동 재활성화 불요). 0=비활성.

watermark(데이터 윈도우 진행 = 무엇을 수집했나), nextScanDueAt(스캔 due = 언제 차례인가), lastScanAttemptAt(관측 커서)은 직교한다.

3.3 A — watermark 증분 + 윈도우 상한 + 롤링 샘플링

watermark 증분은 직전 스캔 이후 경과분(델타)만 수집한다. 여기에 max-window 상한(백필 슬라이스)과 기본 활성화된 롤링 샘플링(D66)·delta-skip(D60)·watermark 점프(D61)가 얹혀 운영 Loki 부하를 실측 안정선까지 낮춘다.

기본(sample-window > 0) window = [ max(lastEnd, end−sample-window) , end ), end = now − ingest-lag
// 항상 '최신 sample 구간'만 조사 — 과거 백로그는 의도적으로 건너뜀(표본화). 신규 도메인도 최신 구간만(백필 미발동)
sample-window = 0(gap-free 크롤) window = [ lastEnd (없으면 end−initial-backfill) , end ); if (end−start) > max-window → end = start+max-window
  • 롤링 샘플링(D66, sample-window=PT10M 기본): 주기 스캔은 최신 sample-window 표본만 본다. 과거 워터마크~시작 사이 백로그는 의도적으로 skip → 대량 백필 발산 정지. 신선도 = 재방문 주기(active 30분 = 시간당 2회 표본). 0=끄면 아래 gap-free 크롤(백필)로 동작.
  • delta-skip(D60): 디스커버리의 마지막 관측(lastSeenAt)이 윈도우 시작 이전이면 = 구간에 신규 트래픽 없음 → Loki 조회 없이 watermark 만 전진(빈 윈도우 쿼리 낭비 제거). 대부분의 도메인이 매 틱 이 경로로 값싸게 skip 된다.
  • watermark 점프(D61): delta-skip 시 watermark 를 now−ingest-lag 까지 즉시 전진(max-window 미적용) → 트래픽 재개 시 빈 과거 윈도우를 기어가는 낭비 없이 near-now 유지.
  • 백필 슬라이스(gap-free 모드): 샘플링을 끄면 미스캔/지연 도메인이 1회 최대 max-window(PT30M)만 수집하고 watermark 가 그만큼 전진 → 여러 틱 점진 백필.
  • watermark 전진: 성공적으로 소비한 구간 뒤에만(at-least-once, 부분 소비 시 마지막 완료 슬라이스 끝). nextWindow empty면 skip.
  • 온디맨드 무관: -domain -scan 온디맨드는 샘플링/delta-skip/점프를 모두 우회하고 임시 윈도우만 본다(watermark 미전진, 4.3).

3.4 E — 전역 레이트 가드 + 자동감속

LokiClient 의 기존 보호(동시 2 · 200ms 스로틀 · page-limit 2000 · 백오프) 위에 전역 예산과 적응형 감속을 둔다 — 이것이 Loki 부하의 하드 천장이다.

메커니즘동작
LokiBudget 전역 시간당 하드캡현재 시간창의 누적 쿼리 수 + 응답 바이트를 카운트(시간 롤오버 리셋). max-queries-per-hour/max-bytes-per-hour 초과 시 hasBudget()=false → 틱 조기 종료(이월). 0=무제한.
적응형 throttle (throttle-on-error)429/5xx 발생 시 inter-query 간격을 min-query-interval × 2^level 로 키움(level 0→4, 최대 16×), 연속 성공 시 감쇠. 일시적 per-attempt 백오프에 더해 지속 슬로다운.
Micrometer 계측Actuator /metrics 로 노출: loki.queries(쿼리 수) · loki.response.bytes(응답 바이트 근사) · loki.errors{status}(429/5xx 및 타임아웃·연결실패 등 I/O 실패 — I/O 실패도 1쿼리로 시간당 예산에 계상해 느린 Loki hammering 을 예산으로 제어, D58).

정확도 한계(정직): 바이트는 응답 본문 길이 근사(헤더/압축 제외), 시간창 경계 카운트는 ±1틱 오차 — 운영 가시화엔 충분하나 정밀 과금용은 아니다.

4. 운영자 온디맨드 수집 CLI 구현됨

근거: doc/33 §7(scan·register) · doc/31 §B(export) · DECISIONS D45 / D47 / D43. 트리거 인자가 있으면 main() 이 웹 미기동 + cli 프로파일(스케줄러 미기동)로 1 명령 실행 후 종료.

스케줄을 기다리지 않고 특정 도메인을 즉시 조회·등록·수집·내보낼 때 쓴다. 네 서브커맨드 모두 -domain 아래 단일대시 문법이다(트리거 인자가 있으면 CLI 모드로 1 명령 실행 후 종료).

복수 서브커맨드·도메인 누락·-domain 단독은 usage 에러(exit 2)다.

컨테이너 실행 방식 — exec vs run
  • 실행 중 컨테이너(adc-app)가 있으면 scan·register 는 exec 로(새 컨테이너·볼륨 불필요 — 둘 다 파일 산출이 없다. scan 은 DB 영속+요약, register 는 DB 등록만). exec 는 ENTRYPOINT(wait-for-db.sh)를 우회하므로 java -jar /app/app.jar 를 명시한다. one-off run 으로 쓸 때도 -v 는 불요.
  • list·export 는 run -v 로. 둘 다 CSV 파일을 산출하는데 서빙 pod 에는 /exports 가 마운트돼 있지 않아(CSV 는 PGDATA 밖) host 로 파일을 빼려면 볼륨을 단 새 one-off 컨테이너가 필요하다.
  • 컨테이너 미가동 상태에선 어느 명령이든 one-off podman run --rm --network host … <명령> 으로 실행한다(인자는 ENTRYPOINT 뒤 append).

4.1 도메인 목록 — -domain -ls

# /exports 볼륨을 단 one-off 컨테이너로 실행 (서빙 pod 엔 /exports 미마운트)
podman run --rm --network host -v /opt/adc-exports:/exports \
  localhost/apidiscover:test -domain -ls
# → /exports/domains-<epochmillis>.csv  (호스트의 /opt/adc-exports 에 영속)
  • DB 에 등록된 전체 도메인을 CSV 파일로 내보낸다(-export 와 동일 다운로드 패턴). Loki 미호출(DB only)이라 부하가 없다.
  • CSV 헤더: host,enabled,hostnames,discovered_at,last_seen_at. hostnames 는 여러 개면 ; 로 join, null 값은 공란(RFC4180·CRLF).
  • 출력 디렉터리 기본 /exports(adc.cli.output-dir). 빈 목록이면 헤더만 있는 CSV 가 생성된다.
  • 종료 코드: 0 성공(빈 목록=헤더만 CSV) / 4 DB·쓰기 오류.

4.2 도메인 즉시 등록 — -domain -register

# 파일 산출이 없어 -v(볼륨) 불요 — one-off run 또는 실행 중 컨테이너 exec
podman run --rm --network host localhost/apidiscover:test \
  -domain -register <도메인>
# → 정규화 host 로 enabled=true 등록 ("이미 등록됨"=no-op·exit 0)
  • 도메인을 DB 에 등록만 한다(Loki 미호출·부하 없음). 스캔·디스커버리는 하지 않는다.
  • host 는 정규화(소문자·trim)해 저장하며 멱등이다 — 이미 있으면 아무것도 바꾸지 않고 성공(exit 0)으로 끝난다.
  • 자동 디스커버리(interval=PT10M)가 아직 잡지 못한 신규 도메인을 미리 등록해 두는 용도. 등록 후 다음 스캔 틱(due 도래 시)에서 자동 수집된다.
  • 종료 코드: 0 성공(이미 존재 포함) / 2 도메인 미지정 / 4 DB 오류.
운영자 즉시 처리 워크플로 — 신규 도메인 자동 경로는 디스커버리(interval=PT10M 주기)가 도메인을 잡은 뒤 다음 스캔 틱에서 수집되므로 시간이 걸린다. 급하면 즉시 경로를 쓴다(이 CLI 로 수 초~수 분, 윈도우·도메인 규모에 따라 변동).
  • 한 번에 등록+검출 — -domain -scan <도메인> 한 번이면 미등록이어도 자동 등록(enabled=true) 후 즉시 스캔한다(별도 등록 불필요).
  • 등록만 먼저 — -domain -register 로 잡아두면 스케줄러가 다음 틱부터 자동 수집한다(즉시 Loki 부하 없음).

4.3 즉시 API 수집 — -domain -scan

# 실행 중 컨테이너에 exec (권장)
podman exec adc-app java -jar /app/app.jar \
  -domain -scan <도메인> [-window PT1H] [-edge <hostname>]

# 출력 예 (미등록 도메인이면 자동 등록 줄이 먼저 나온다)
# 자동 등록: example.com (enabled=true)
# scan example.com [2026-06-26T05:30:00Z ~ 2026-06-26T06:00:00Z]
#   discovered=128 active=40 shadow=12 zombie=3 unused=7
  • 미등록 도메인이면 enabled=true 로 자동 등록한 뒤 스캔한다(별도 -register 불필요). 이미 존재하지만 비활성(enabled=false)이면 운영자 결정을 존중해 자동 활성화하지 않고 스캔을 거부한다(exit 3).
  • -window 미지정 시 기본/상한 = scan.max-window. -edge 미지정 시 도메인의 hostnames 전체를 수집, 지정 시 그 엣지만.
  • 결과는 ScanResult(최신 결과) + discovered_endpoint(누적 카탈로그)에 반영된다.
★ watermark 미전진 — 스케줄 갭 없음 온디맨드 스캔은 임시 윈도우만 보고 watermark 를 전진시키지 않는다. 만약 즉시 스캔이 [now−1h, now) 를 보고 watermark 를 당기면, 스케줄러가 그 사이 구간 [lastEnd, now−1h) 를 건너뛰어 데이터 갭이 생긴다. 그래서 온디맨드는 누적/최신 결과만 갱신하고, 증분 진행의 소유권은 스케줄러가 계속 가진다.

5. 운영 가이드 — 부하 · 모니터링 · 튜닝 · 트러블슈팅

5.1 운영 Loki 부하 관리 원칙

  • 대상 Loki(192.168.8.100:3200)는 운영 서버다. LokiClient 의 부하보호(윈도우 분할·page-limit·동시성·스로틀·백오프)를 우회하지 말 것. 임시 스크립트도 창/limit 을 작게.
  • 가장 강한 안전판은 scan.max-queries-per-scan(도메인 1회)·scan.max-queries-per-hour(전역 시간당) 하드캡이다. 부하 불확실 시 보수적으로 시작.
  • off-peak(01:00–06:00, schedule.off-peak-window·scan.off-peak-zone=Asia/Seoul): 자동 배선 완료(OffPeakWindow 가 scanTick/ScanSelector 에서 K·윈도우 스위치). ★단 현재는 off-peak 값을 주간과 동일하게 튜닝(off-peak-max-window=PT30M·off-peak-domains-per-tick=500) — 과거 대량 조회(PT24H)가 Loki 를 과부하시킨 이력 때문. 대량 백필이 필요하면 이 값을 신중히 올리고 off-peak 시간대에 한정하라.

5.2 모니터링 (Micrometer / Actuator)

Actuator /metrics(노출: health,info,metrics)에서 Loki 호출량을 관찰한다.

loki.queries시간 경과 누적 Loki 쿼리 수 — 시간당 캡 대비 사용률 추적 loki.response.bytes응답 바이트 근사 — 대량 응답 도메인 탐지 loki.errors{status}429/5xx 카운트(status 태그별) — Loki 불안정·throttle 작동 신호

메트릭 임계 기반 알람/통지 연동은 예정(현재는 메트릭 노출까지). loki.errors 증가 시 적응형 throttle 가 자동으로 속도를 낮춘다.

도메인 급증(디스커버리가 많이 등록)커버리지가 느려지면 domains-per-tick ↑ 또는 tick-interval ↓. 단 max-queries-per-hour 를 함께 점검해 시간당 부하 상한 유지. (등록 자체는 Loki 부하와 무관.)
Loki 부하 경보 / 429 빈발throttle-on-error=true 가 자동감속(이미 기본). 추가로 max-queries-per-hour 를 일시 하향, domains-per-tick 하향. loki.errors 추세 확인.
특정 도메인 결과가 급히 필요스케줄을 기다리지 말고 온디맨드 CLI(-domain -scan)로 즉시 스캔(4장). watermark 미전진이라 스케줄 갭 없음.
신규(미등록) 도메인을 즉시 검출디스커버리(interval=PT10M)를 기다리지 말고 -domain -scan 한 번으로 자동등록+즉시 스캔(4.3). 등록만 미리 해두려면 -domain -register(4.2).
과거 데이터가 필요(샘플링이 백로그 skip)기본 sample-window=PT10M 은 최신 표본만 본다. 과거 구간이 필요하면 sample-window=0(gap-free 크롤)로 두고 off-peak-max-window ↑ 로 off-peak 백필 가속(대량 조회는 Loki 과부하 주의). 활동 티어링(C)·off-peak(D)은 이미 기본 활성.

6. 로드맵 — 구현됨 vs 예정

오해 방지를 위해 현재 동작과 예정 기능을 명확히 구분한다. B·A·E 부하 정책과 C·D·F 티어링, 그리고 운영 튜닝(D56~D69)까지 모두 머지·배포 검증 완료다.

단계내용상태
B+A+E (D45)틱당 예산+due 선택(B) · watermark 증분+윈도우 상한(A) · 전역 레이트 가드+자동감속+Micrometer(E) · 온디맨드 스캔/내보내기 CLI구현·머지 완료
C+D+F (D48)C 활동 티어링(active/inactive/default 주기 + intervalOverride 배선) · D off-peak 스위치(OffPeakWindow 배선, K/윈도우 주입) · F dormant 디프라이오리티(장기 무트래픽 최장 주기 강등, 삭제 아님) — 단일 due 모델로 통합구현·머지 완료
운영 튜닝 (D56~D69)무접속 중단(inactive-after) · 롤링 샘플링(sample-window) · delta-skip/watermark 점프 · 엣지 배칭(query-batch-size) · 엣지 그룹 Master(edge-group-main-only) · 대상 제외 엣지(excluded-hostnames)구현·배포 검증
메트릭 알람Micrometer 메트릭 임계 기반 알람/통지 연동(현재는 메트릭 노출까지)예정
