2. ① 도메인 수집 (디스커버리) 구현됨 · PR1

근거: doc/30 · DECISIONS D42 / D44. 담당 컴포넌트: DomainDiscoveryScheduler · DomainDiscoveryService · DomainUpserter · LokiClient(instant 쿼리).

수집 중 access log 에서 API 도메인(Host 헤더) 을 자동 열거해 DomainConfig 를 채운다. 수동 등록 부담을 없애는 것이 목적이다. 핵심은 원시 로그를 받지 않고 Loki 서버측에서 집계한 (도메인 × 엣지 hostname) 카운트만 수신한다는 점 — 그래서 매우 가볍다.

2.1 동작 방식 — 서버측 집계 쿼리

도메인은 Loki 라벨이 아니라 로그 라인 내용(필드 15 host / 16 real_host, 구분자 ^|^)에 있다. 따라서 서버에서 라인을 pattern 으로 파싱·집계하되, 도메인 합성(coalesce)·필터·검증은 클라이언트(Java) 에서 한다.

sum by (host, real_host, hostname) (
  count_over_time(
    {job="access_log"}
      | pattern "<_>^|^<_>^|^ ... (인덱스 0–14 skip) ... ^|^<host>^|^<real_host>^|^<_>"
    [W]
  )
)
# instant 벡터 쿼리(/loki/api/v1/query) — 응답은 집계 1벡터(라인 미전송)
★ coalesce 는 반드시 클라이언트에서 (실측 교훈, D44) 서버측 | label_format domain=coalesce(...) 를 붙이면 Go 템플릿이 라인마다 평가돼 약 10배 느려진다(실 Loki 5분창 2.2s → 20.2s) → 부트스트랩 1h·롤링 12m 모두 query-timeout(30s) 초과 → 디스커버리 실패. 그래서 서버는 host·real_host·hostname 으로만 group by 하고, 클라이언트가 firstNonEmpty(normalizeDomain(host), normalizeDomain(real_host)) 로 합성한다(LogLineParser 와 동일 의미: host 빈/- → real_host 폴백).

같은 도메인이 여러 (host, real_host)·여러 hostname 으로 나오면 클라이언트가 coalesce 후 도메인 키로 합산(hostnames 합집합·count 합)해 DomainConfig.hostnames(엣지 서버 집합)를 자동으로 채운다. 이 hostnames 가 ②스캔에서 어느 엣지를 조회할지 결정한다.

2.3 주기 · 설정 키

스캔과 별도 스케줄러(DomainDiscoveryScheduler)로 돌며, initial-delay 로 스캔과 시작 시점을 어긋나게(stagger) 해 동시 Loki 타격을 피한다. 첫 실행(빈 도메인 DB)만 bootstrap-window(1h) 로 초기 도메인을 일괄 확보하고 이후 롤링한다.

3. ② 도메인별 API 정보 수집 (스캔) 구현됨 · PR1

근거: doc/33 · DECISIONS D45(PR1 = B+A+E). 담당: DiscoveryScheduler.scanTick() · ScanSelector · DiscoveryJobService · LokiBudget · LokiClient.

3.1 왜 부하 정책이 필요한가

디스커버리가 ~13,920+ 도메인을 무제한 등록하므로, 과거처럼 @Scheduled(PT1H) 로 enabled 전 도메인을 전수 순차 스캔하면 운영 Loki 가 지속 과부하된다. 특히 미스캔 도메인의 첫 윈도우는 end − initial-backfill(P7D) = 7일치라, 7일/10분 ≈ 1008 쿼리 × hostnames를 한 번에 당긴다(14k 동시 백필 = 재앙).

그래서 현재 스캐너는 전수 순회를 슬라이스 순회로 바꾸고, 세 가지 축으로 부하를 통제한다 — B(틱당 예산), A(윈도우 상한), E(전역 하드캡).

3.2 B — 틱당 예산 + 라운드로빈(least-recently-scanned)

한 틱에 전 도메인을 금지하고, 짧은 틱마다 예산(K) 내에서 "가장 오래 안 본 도메인부터" 처리한다.

  • 선택: ScanSelector.selectForTick() = DomainConfig.lastScanAttemptAt 오름차순(nulls first = 미스캔 우선) 상위 domains-per-tick(K) 개.
  • 커서 = 영속 타임스탬프: lastScanAttemptAt(재기동 생존). 메모리 인덱스 미채택.
  • ★커서는 attempt 마다 전진(skip·실패 포함). watermark 가 최신이라 runScan 이 조기 skip 해도 큐 뒤로 회전 → up-to-date 도메인을 매 틱 재선택하는 예산 낭비 방지.
  • 무기아: lastScanAttemptAt asc = FIFO → 최악 대기 = 전수 1회 커버리지 시간(유한).

watermark(데이터 윈도우 진행 = 무엇을 수집했나)와 lastScanAttemptAt(스케줄 공정성 커서 = 언제 차례를 줬나)은 직교한다(둘 다 필요).

3.3 A — watermark 증분 + per-scan 윈도우 상한

watermark 증분(이미 동작)은 정상상태에서 직전 스캔 이후 경과분(델타)만 수집한다. 여기에 max-window 상한을 더해 백필을 슬라이스한다.

window = [ lastEnd (없으면 end−initial-backfill) , now − ingest-lag )
if (end − start) > max-window → end = start + max-window // 7일 일괄 pull 차단, 여러 틱에 점진 백필
  • 정상상태: window = 직전 이후 경과분(자연히 작음, 보통 틱 간격 수준) → 상한 미발동.
  • 백필: 미스캔/지연 도메인은 1회 최대 max-window(PT6H)만 수집하고 watermark 가 그만큼 전진 → 여러 틱에 걸쳐 점진 백필.
  • watermark 전진: 성공 후에만(at-least-once). nextWindow 가 empty면 skip.
  • staleness(정직): 저빈도 도메인은 사이클당 1회·max-window 로만 전진하므로 실시간 대비 ~사이클 시간 뒤처질 수 있다 — 엔드포인트 디스커버리엔 허용(실시간 경보 아님), off-peak(예정)로 가속.

3.4 E — 전역 레이트 가드 + 자동감속

LokiClient 의 기존 보호(동시 2 · 200ms 스로틀 · page-limit 2000 · 백오프) 위에 전역 예산과 적응형 감속을 둔다 — 이것이 Loki 부하의 하드 천장이다.

메커니즘동작
LokiBudget 전역 시간당 하드캡현재 시간창의 누적 쿼리 수 + 응답 바이트를 카운트(시간 롤오버 리셋). max-queries-per-hour/max-bytes-per-hour 초과 시 hasBudget()=false → 틱 조기 종료(이월). 0=무제한.
적응형 throttle (throttle-on-error)429/5xx 발생 시 inter-query 간격을 min-query-interval × 2^level 로 키움(level 0→4, 최대 16×), 연속 성공 시 감쇠. 일시적 per-attempt 백오프에 더해 지속 슬로다운.
Micrometer 계측Actuator /metrics 로 노출: loki.queries(쿼리 수) · loki.response.bytes(응답 바이트 근사) · loki.errors{status}(429/5xx).

정확도 한계(정직): 바이트는 응답 본문 길이 근사(헤더/압축 제외), 시간창 경계 카운트는 ±1틱 오차 — 운영 가시화엔 충분하나 정밀 과금용은 아니다.

하드 천장이 안전판 어떤 노브를 만지든 max-queries-per-hour(와 선택적 max-bytes-per-hour)가 시간당 Loki 호출의 상한을 강제한다. 부하가 걱정되면 이 값부터 보수적으로 잡고 커버리지를 점진 확대하라.

4. 운영자 온디맨드 수집 CLI 구현됨 · PR1

근거: doc/33 §7(scan) · doc/31 §B(export) · DECISIONS D45 / D43. 트리거 인자가 있으면 main() 이 웹 미기동 + cli 프로파일(스케줄러 미기동)로 1 명령 실행 후 종료.

스케줄을 기다리지 않고 특정 도메인을 즉시 조회·수집·내보낼 때 쓴다. 세 서브커맨드 모두 -domain 아래 단일대시 문법이다(트리거 인자가 있으면 CLI 모드로 1 명령 실행 후 종료).

서브커맨드동작Loki
-domain -ls등록 도메인 목록무관(DB only)
-domain -export <도메인>결과 CSV 내보내기무관(DB only)
-domain -scan <도메인> [-window <ISO8601>] [-edge <hostname>]온디맨드 즉시 스캔운영 Loki 호출 · watermark 미전진

복수 서브커맨드·도메인 누락·-domain 단독은 usage 에러(exit 2)다.

컨테이너 실행 방식 — exec vs run
  • 실행 중 컨테이너(adc-app)가 있으면 list·scan 은 exec 로(새 컨테이너 불필요). exec 는 ENTRYPOINT(wait-for-db.sh)를 우회하므로 java -jar /app/app.jar 를 명시한다.
  • export 는 run -v 로. 서빙 pod 에는 /exports 가 마운트돼 있지 않아(CSV 는 PGDATA 밖) host 로 파일을 빼려면 볼륨을 단 새 one-off 컨테이너가 필요하다.
  • 컨테이너 미가동 상태에선 어느 명령이든 one-off podman run --rm --network host … <명령> 으로 실행한다(인자는 ENTRYPOINT 뒤 append).

4.1 도메인 목록 — -domain -ls

podman exec adc-app java -jar /app/app.jar -domain -ls
  • DB 에 등록된 도메인을 열거한다. Loki 미호출(DB only)이라 부하가 없다.

4.2 즉시 API 수집 — -domain -scan

# 실행 중 컨테이너에 exec (권장)
podman exec adc-app java -jar /app/app.jar \
  -domain -scan <도메인> [-window PT1H] [-edge <hostname>]

# 출력 예
# scan example.com [2026-06-26T00:00:00Z ~ 2026-06-26T06:00:00Z]
#   discovered=128 active=40 shadow=12 zombie=3 unused=7
  • -window 미지정 시 기본/상한 = scan.max-window. -edge 미지정 시 도메인의 hostnames 전체를 수집, 지정 시 그 엣지만.
  • 결과는 ScanResult(최신 결과) + discovered_endpoint(누적 카탈로그)에 반영된다.
★ watermark 미전진 — 스케줄 갭 없음 온디맨드 스캔은 임시 윈도우만 보고 watermark 를 전진시키지 않는다. 만약 즉시 스캔이 [now−1h, now) 를 보고 watermark 를 당기면, 스케줄러가 그 사이 구간 [lastEnd, now−1h) 를 건너뛰어 데이터 갭이 생긴다. 그래서 온디맨드는 누적/최신 결과만 갱신하고, 증분 진행의 소유권은 스케줄러가 계속 가진다.

4.3 결과 내보내기(CSV) — -domain -export

# /exports 볼륨을 단 one-off 컨테이너로 실행 (서빙 pod 엔 /exports 미마운트)
podman run --rm --network host -v /opt/adc-exports:/exports \
  localhost/apidiscover:test -domain -export <도메인>
# → /exports/<도메인>-<epochmillis>.csv  (호스트의 /opt/adc-exports 에 영속)
  • 결합 Discovery 결과(discovered_endpoint ∪ 스펙)를 CSV 로 출력. 출력 디렉터리 기본 /exports(adc.cli.output-dir).
  • CSV 에 내용이 나오려면 먼저 그 도메인이 스캔되어 discovered_endpoint 가 채워져 있어야 한다. 갓 디스커버리된 도메인은 다음 스캔 전까지 "검출 0건"(exit 3)이 정상.
  • 즉시 스캔 후 CSV 가 필요하면 scan → export 를 순차 2회 호출(단일 체인 플래그는 미제공·불요).

5. 운영 가이드 — 부하 · 모니터링 · 튜닝 · 트러블슈팅

5.1 운영 Loki 부하 관리 원칙

  • 대상 Loki(192.168.8.100:3200)는 운영 서버다. LokiClient 의 부하보호(윈도우 분할·page-limit·동시성·스로틀·백오프)를 우회하지 말 것. 임시 스크립트도 창/limit 을 작게.
  • 가장 강한 안전판은 scan.max-queries-per-hour 시간당 하드캡이다. 부하 불확실 시 보수적으로 시작.
  • off-peak(01:00–06:00): schedule.off-peak-window 에 정의돼 있으나 현재는 설정값만 존재하고 자동 배선은 예정 · PR2(D). 그 전까지 대용량 백필/확인 작업은 운영자가 해당 시간대에 수동(CLI)으로 수행하는 것을 권장.

5.2 모니터링 (Micrometer / Actuator)

Actuator /metrics(노출: health,info,metrics)에서 Loki 호출량을 관찰한다.

loki.queries시간 경과 누적 Loki 쿼리 수 — 시간당 캡 대비 사용률 추적 loki.response.bytes응답 바이트 근사 — 대량 응답 도메인 탐지 loki.errors{status}429/5xx 카운트(status 태그별) — Loki 불안정·throttle 작동 신호

메트릭 임계 기반 알람/통지 연동은 예정(현재는 메트릭 노출까지). loki.errors 증가 시 적응형 throttle 가 자동으로 속도를 낮춘다.
