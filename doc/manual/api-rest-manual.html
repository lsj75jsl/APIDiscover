REST API 사용 매뉴얼

API Discovery Worker 가 제공하는 REST 엔드포인트 — 도메인 관리·스캔·결과·판단 근거·스펙·분류 설정 사용법과 실제 응답 예시.

Base URL http://<host>:8080/api/v1 · 응답 application/json · 예시 호스트 api.weble.net

Spring Boot 3.3 전 응답 라이브 캡처 ⚠️ 인증 없음 — §4 제약 필독
⚠️ 인증·인가가 없다 현재 모든 엔드포인트는 인증이 없다 — 8080 포트에 네트워크로 도달 가능한 누구나 호출할 수 있다. 운영 검증용 상태이며, 외부 노출 전 반드시 방화벽 IP 제한·게이트웨이 뒤 배치 등 보호가 필요하다(자세한 내용 §4 제약사항 · §5 TODO).

1. 개요 · 엔드포인트 한눈에 보기

모든 도메인 API 는 http://<host>:8080/api/v1 아래에 있고, 헬스체크 등 운영 엔드포인트는 /actuator 아래에 있다. 응답은 JSON 이다. 아래 표의 ⚠️ Loki 표시가 있는 두 엔드포인트만 운영 Loki 를 호출(부하 발생)하며 나머지는 전부 DB 조회다.

메서드경로용도Loki
GET/domains전체 도메인 목록(대량·N+1 주의)DB
POST/domains도메인 등록(201)DB
GET/domains/{host}도메인 설정 조회DB
PUT/domains/{host}도메인 설정 수정DB
DELETE/domains/{host}도메인 삭제(204)DB

POST/domains/{host}/scan온디맨드 재스캔(202)⚠️ Loki
GET/domains/{host}/discovery결합 디스커버리 + 판단 근거DB
PUT/domains/{host}/spec스펙 업로드DB
GET/domains/{host}/spec활성 스펙 메타DB

아래 실제 응답은 라이브 배포(전역 도메인 42,809건)에서 캡처한 것이다. 예시 도메인 api.weble.net 은 api 서브도메인이며 스펙이 없어 검출이 전부 Shadow 다.

2. 엔드포인트 상세 (사용법 · 실제 응답)

2.1 도메인 인벤토리 — /api/v1/domains

GET/api/v1/domains

등록된 전체 도메인 목록. 각 원소는 GET /domains/{host} 와 동일한 객체(DomainView)다.

curl http://<host>:8080/api/v1/domains
[
  { "host":"www.takigen.co.jp", "enabled":true, "hostnames":["..."],
    "intervalOverride":null, "specMergeStrategy":"MERGE", "basePathStrip":null, "spec":null },
  { "host":"api.weble.net", "enabled":true, "hostnames":["AORV1","AORV2","PORV1","PORV2"],
    "intervalOverride":null, "specMergeStrategy":"MERGE", "basePathStrip":null, "spec":null }
  /* … 총 42,809건 (페이지네이션 없음) */
]
대량 주의 42,809건을 한 번에 반환하고 도메인마다 활성 스펙을 조회(N+1)하므로 느리고 무겁다. 대량 환경에서는 비권장 — 페이지네이션은 TODO.
POST/api/v1/domains

도메인 등록. host 는 정규화(소문자·trim)되며 이미 있으면 409 Conflict. 성공 시 201 Created + 등록된 DomainView. (CLI -domain -register 로도 등록 가능.)

curl -X POST http://<host>:8080/api/v1/domains \
  -H 'Content-Type: application/json' \
  -d '{"host":"api.example.com","enabled":true}'

요청 body(DomainUpsert) 선택 필드: hostnames·intervalOverride·specMergeStrategy·basePathStrip.

GET/api/v1/domains/{host}

한 도메인의 설정. 엣지 hostnames·병합 전략·base-path-strip·활성 스펙 메타(spec)를 담는다.

curl http://<host>:8080/api/v1/domains/api.weble.net
{
  "host":"api.weble.net",
  "enabled":true,
  "hostnames":["AORV1","AORV2","PORV1","PORV2"],
  "intervalOverride":null,
  "specMergeStrategy":"MERGE",
  "basePathStrip":null,
  "spec":null
}
PUT/api/v1/domains/{host}  ·  DELETE/api/v1/domains/{host}

PUT 은 설정 수정(body = DomainUpsert, GET 응답과 동형 반환). DELETE 는 삭제 후 204 No Content(없으면 404).

# 설정 수정 — 예: 병합 전략·스캔 주기 override
curl -X PUT http://<host>:8080/api/v1/domains/api.weble.net \
  -H 'Content-Type: application/json' \
  -d '{"host":"api.weble.net","enabled":true,"specMergeStrategy":"MERGE","intervalOverride":"PT1H"}'

# 삭제
curl -X DELETE http://<host>:8080/api/v1/domains/api.weble.net   # → 204

2.2 스캔 상태 · 결과 — /scan-status · /result · /scan

GET/api/v1/domains/{host}/scan-status

가벼운 상태 메타. 중앙 서버가 version·lastScanAt 만 비교해 결과 재조회 여부를 결정한다. summary 카운트 의미는 §2.3·DB 스키마 매뉴얼 참조.

curl http://<host>:8080/api/v1/domains/api.weble.net/scan-status
{
  "host":"api.weble.net",
  "state":"idle",
  "lastScanAt":"2026-06-26T09:04:13.072008Z",
  "version":"1913dd805b705e7a",
  "summary":{"discovered":166,"active":0,"shadow":161,"zombie":0,"unused":0},
  "totalDropped":5
}
GET/api/v1/domains/{host}/result

스캔 전체 리포트(DiscoveryReport JSON). 조건부 GET 대상 — ETag 응답 헤더에 version 이 실린다(§3). 아직 완료된 스캔이 없으면 204 No Content.

curl -i http://<host>:8080/api/v1/domains/api.weble.net/result

응답 본문 상위 키와 핵심 부분(트림).

{
  "host":"api.weble.net", "generatedAt":"…", "logWindow":{…}, "specVersion":…,
  "summary":{"discovered":166,"active":0,"shadow":161,"zombie":0,"unused":0,"lowConfidence":0},
  "findings":[
    { "host":"api.weble.net", "method":"GET", "pathTemplate":"/requests",
      "confidence":0.9, "reason":"트래픽 존재, 문서 내 매칭 템플릿 없음",
      "params":{ "query":[
          {"name":"campaignId","count":10075,"lenBuckets":["S"],"sensitive":false},
          {"name":"type","count":10075,"lenBuckets":["S"],"sensitive":false} ],
        "path":[] },
      "low_confidence":false }
  ],
  "droppedNonApi":{"excluded":0,"webForm":0,"lowScore":5,"total":5},
  "droppedByLimit":{…}, "droppedNonExistent":{…},
  "endpointKindSignal":{…}, "typeDistribution":{…}, "preflightSignal":{…}, "specSource":{…}
}

상위 키: host·generatedAt·logWindow·specVersion·summary·findings·droppedNonApi·droppedByLimit·droppedNonExistent·endpointKindSignal·typeDistribution·preflightSignal·specSource.

POST/api/v1/domains/{host}/scan ⚠️ Loki

온디맨드 재스캔 트리거. 운영 Loki 를 호출하므로 남발 금지. 즉시 202 Accepted 를 반환한다(현재는 동기 트리거 — 비동기 큐잉은 TODO).

curl -i -X POST http://<host>:8080/api/v1/domains/api.weble.net/scan   # → 202 Accepted

2.3 결합 디스커버리(판단 근거) — /discovery

GET/api/v1/domains/{host}/discovery

검출 카탈로그와 스펙을 결합한 결과다. 일반 findings 외에 도메인의 현재 설정(effectiveClassification)과 엔드포인트별 판단 근거(rationale)가 함께 내려온다(신규 기능). 신호·가중치를 읽는 자세한 법은 API 판독 매뉴얼 §4.3 을 참조한다.

curl http://<host>:8080/api/v1/domains/api.weble.net/discovery

① 도메인 현재 설정 — effectiveClassification

"effectiveClassification":{
  "profile":"MIDDLE",
  "threshold":0.7,
  "weightsSource":"preset",
  "weights":{
    "hostApiSubdomain":0.4, "corsPreflight":0.3, "apiSegment":0.55, "graphqlSegment":0.55,
    "versionSegment":0.26, "pathIdSegment":0.15, "machineEndpoint":0.2, "writeMethod":0.34,
    "query":0.12, "nonBrowserUa":0.24, "staticAssetPenalty":-0.6, "repeatBonus":0.12,
    "pathHint":0.55, "responseTypeApi":0.25
  }
}

② 엔드포인트별 판단 근거 — rationale[] (findings 와 병렬)

rationale[i] 는 findings[i] 와 동일 순서·동일 identity(method·host·pathTemplate)다. 아래는 점수로 판정된 Shadow 의 실제 근거다(api.weble.net, findings·rationale 각 192건).

{
  "method":"POST", "host":"api.weble.net", "pathTemplate":"/qnas",
  "classification":"SHADOW",
  "basis":{
    "type":"score", "apiScore":1.0, "threshold":0.7, "gate":"ADMIT", "mode":"pathless",
    "signals":[
      {"key":"hostApiSubdomain","weight":0.4, "fired":true, "contribution":0.4},
      {"key":"corsPreflight",   "weight":0.3, "fired":true, "contribution":0.3},
      {"key":"writeMethod",     "weight":0.34,"fired":true, "contribution":0.34},
      {"key":"repeatBonus",     "weight":0.12,"fired":true, "contribution":0.12}
      /* … 평가된 신호 전부(13개) — 미발화 신호는 "fired":false,"contribution":0.0 */
    ]
  }
}
읽는 법 — 위 예시 발화 신호 기여 합 0.4 + 0.3 + 0.34 + 0.12 = 1.16 → [0,1] clamp = apiScore 1.0 ≥ threshold 0.7 → gate ADMIT(API 로 확정 → 스펙에 없으므로 Shadow). mode:"pathless" 는 내장 path-shape 모드(operator 힌트 미설정). contribution 을 전부 더하면 apiScore 가 된다.

③ 분류별 근거가 다르다 — basis.type

점수(score)는 스펙에 없는 Shadow 에만 쓰인다. 스펙에 매칭되는 Active·Zombie 는 점수가 아니라 스펙 매칭이 근거다.

분류basis.type주요 필드근거
SHADOWscoreapiScore·threshold·gate·mode·signals[]점수 게이트 통과(ADMIT)
ACTIVEspec_matchspecRef·deprecated:false스펙 매칭(점수 아님)
ZOMBIEspec_matchspecRef·deprecated:true 또는 estimated:true스펙 매칭(deprecated/버전추정)
UNUSEDspec_onlyspecRef스펙에 있고 무트래픽
WEBPAGEendpoint_kindendpointKind·kindConfidence$type/referer 로 비-API(현재 미산출)

근거 설계 doc/34 §2. 비-API 로 drop 된 엔드포인트는 findings·rationale 에 없다(집계 droppedNonApi 만).

2.4 스펙 — /api/v1/domains/{host}/spec

PUT/api/v1/domains/{host}/spec  ·  GET/api/v1/domains/{host}/spec

PUT 은 스펙 원문(OpenAPI / Postman / CSV 바이트)을 업로드한다. GET 은 활성 스펙 메타를 반환하며, 스펙이 없으면 404.

# 업로드(예: OpenAPI YAML)
curl -X PUT http://<host>:8080/api/v1/domains/api.example.com/spec \
  -H 'Content-Type: application/octet-stream' \
  --data-binary @openapi.yaml

# 조회 — 스펙 미업로드 도메인(api.weble.net)
curl -i http://<host>:8080/api/v1/domains/api.weble.net/spec
{ "status":404, "error":"Not Found", "path":"/api/v1/domains/api.weble.net/spec" }

api.weble.net 은 스펙이 없어 모든 검출이 Shadow 로 분류된다(매칭 대조 대상 부재).

2.5 분류 설정 — /classification

GET/api/v1/classification  ·  PUT/api/v1/classification

전역 분류 기본값(profile·임계·가중치·힌트). PUT 으로 수정한다. 도메인이 따로 override 하지 않으면 이 값을 상속한다.

curl http://<host>:8080/api/v1/classification
{
  "profile":"MIDDLE",
  "thresholdOverride":null,
  "customWeights":null,
  "matcher":null,
  "updatedAt":"2026-06-25T14:09:40.440546Z"
}
GET/api/v1/domains/{host}/classification  ·  PUT/api/v1/domains/{host}/classification

도메인별 override. 응답은 override(그 도메인이 명시한 값, 없으면 전부 null)와 effective(상속까지 합산된 실효 설정)를 함께 준다. PUT 으로 도메인 override 를 설정한다.

curl http://<host>:8080/api/v1/domains/api.weble.net/classification
{
  "host":"api.weble.net",
  "override":{ "profile":null, "thresholdOverride":null, "customWeights":null, "matcher":null },
  "effective":{
    "profile":"MIDDLE",
    "weights":{ "hostApiSubdomain":0.4, "corsPreflight":0.3, /* …14키 */ "repeatMinCount":3 }
  }
}

override 가 전부 null = 전역(없으면 MIDDLE) 상속. 병합 우선순위(도메인 → 전역 → preset)는 API 판독 매뉴얼 §4.4 참조.

2.6 엣지 hostname 역방향 — /api/v1/hostnames/{hostname}

GET/api/v1/hostnames/{hostname}/domains

특정 엣지 서버(Loki hostname 라벨)가 서빙하는 도메인 목록(host 문자열 배열).

curl http://<host>:8080/api/v1/hostnames/AORV1/domains
["www.61ysw.com","138.002.019.185","www.9longny.com","10.0.246.198","127.0.0.1"]
/* … 총 14건 */
POST/api/v1/hostnames/{hostname}/domains/{domain}/query ⚠️ Loki

특정 (hostname, domain) 1쌍을 즉시 조회·분석한다. 운영 Loki 를 호출한다. 쿼리 파라미터 from·to(ISO-8601)로 윈도우를 지정할 수 있고, 미지정 시 기본 윈도우를 쓴다. 응답은 scan-status 와 같은 ScanStatusView.

curl -X POST 'http://<host>:8080/api/v1/hostnames/AORV1/domains/api.weble.net/query?from=2026-06-26T00:00:00Z&to=2026-06-26T06:00:00Z'

GET/actuator/health

헬스체크(DB 연결 포함). 그 외 info·metrics·configprops 등이 노출된다.

curl http://<host>:8080/actuator/health

3. 조건부 GET (ETag) — 304 실증

중앙 서버는 변경이 없을 때 큰 리포트 본문을 다시 받지 않도록 조건부 GET 을 쓴다. /scan-status 의 version 을 If-None-Match 헤더에 담아 /result 에 보내면, 결과가 그대로면 304 Not Modified(본문 없음)가 온다.

# 1) version 확인
curl -s http://<host>:8080/api/v1/domains/api.weble.net/scan-status | grep version
#   "version":"1913dd805b705e7a"

# 2) 그 version 을 If-None-Match 로 result 에 — 변경 없으면 304
curl -i -H 'If-None-Match: "1913dd805b705e7a"' \
  http://<host>:8080/api/v1/domains/api.weble.net/result
#   HTTP/1.1 304 Not Modified   ← 실측 확인됨 (본문 재전송 회피)

ETag 값은 따옴표를 포함한다("1913dd805b705e7a"). version 이 달라지면(=새 스캔으로 결과 갱신) 200 OK + 새 본문 + 새 ETag 가 온다. /result·/scan-status 의 version 은 동일 ETag 다.

4. 현재 제약사항

구분제약영향 · 완화
인증인증·인가 없음8080 에 도달 가능한 누구나 호출. 완화 = 방화벽 IP 제한·게이트웨이 뒤 배치(현재 운영검증용).
네트워크외부 접근 차단 상태VM 방화벽이 8080 미개방(열림: 5432/9100/9443) → 현재 VM 내부(localhost)에서만. 외부 노출 시 위 무인증과 함께 고려.
목록GET /domains 무페이지네이션 + 도메인별 spec 조회(N+1)42,809건에서 느리고 무겁다. 대량 환경 비권장.
재계산/discovery·/classification 의 effective·rationale 는 조회 시점 현재 설정으로 재계산스캔 당시 값이 아니다. preset 을 바꾸면 즉시 반영(과거 점수 재현 아님).
카탈로그 vs 스캔/discovery 의 findings·rationale 는 누적 검출 카탈로그(discovered_endpoint) 기반특정 스캔 윈도우의 /scan-status·/result 수치와 다를 수 있다(예: discovery 192 vs scan-status shadow 161).
결과 보관/result 리포트는 도메인당 최신 1건만 보관(덮어쓰기)과거 이력 없음.
Loki 부하POST /scan·POST /hostnames/…/query 만 운영 Loki 호출남발 금지(부하보호). 나머지는 DB-only.
dropped비-API 로 drop 된 엔드포인트는 findings·rationale 에 없음집계 droppedNonApi 만 제공.
