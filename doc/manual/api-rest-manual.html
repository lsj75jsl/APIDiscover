2. 엔드포인트 상세 (사용법 · 실제 응답)

2.1 도메인 인벤토리 — /api/v1/domains

GET/api/v1/domains

페이지네이션 도메인 목록. 응답 body 는 JSON 배열(각 원소는 경량 DomainView)이고, 페이지 정보는 응답 헤더에 실린다. 파라미터 ?page(0-based, 기본 0)·?size(기본 1000, 범위 [1,1000]). host 오름차순 정렬(결정적).

전체 도메인이 4만+ 건이라 페이지네이션이 필수다(과거 무페이지 전건 반환은 폐지). X-Total-Pages·X-Current-Page 로 순회한다. 목록의 spec 은 경량 메타이며, 페이지당(최대 1000건) 스펙 조회가 있다(배치 로드는 TODO).

POST/api/v1/domains

도메인 등록. host 는 정규화(소문자·trim)되며 이미 있으면 409 Conflict. 성공 시 201 Created + 등록된 DomainView. (CLI(Command-Line Interface) -domain -register 로도 등록 가능.)

curl -X POST http://<host>:8080/api/v1/domains \
  -H 'Content-Type: application/json' \
  -d '{"host":"api.example.com","enabled":true}'

요청 body(DomainUpsert) 선택 필드: hostnames·intervalOverride·specMergeStrategy·basePathStrip.

PUT/api/v1/domains/{host}  ·  DELETE/api/v1/domains/{host}

PUT 은 부분 수정(PATCH 의미)이다 — body 에 전달한 필드만 반영하고 미전달 필드는 기존값을 유지한다. 전 항목을 다 보낼 필요가 없다. DELETE 는 삭제 후 204 No Content(없으면 404).

# 부분 수정 — enabled 만 비활성으로 (나머지 설정은 그대로 유지)
curl -X PUT http://<host>:8080/api/v1/domains/api.weble.net \
  -H 'Content-Type: application/json' \
  -d '{"enabled":false}'

# 삭제
curl -X DELETE http://<host>:8080/api/v1/domains/api.weble.net   # → 204

미전달 규칙: enabled 미전달=유지(명시 false 만 비활성), hostnames 미전달=유지·[]=비우기(명시 빈 배열), intervalOverride·specMergeStrategy·basePathStrip 미전달=유지. 응답은 수정된 DomainView.

★ enabled ≠ 스캔 (D82)enabled 는 activity_status(무접속 시 자동 강등되는 활동 게이트)와 별개 축이다. 주기 스캔 대상 = enabled=true AND activity_status='ACTIVE' — enabled 를 켜도 무접속으로 INACTIVE 상태면 스캔되지 않는다. 즉시 스캔·주기 재포함은 §2.2 /scan-now 또는 트래픽 재개(다음 discovery 가 ACTIVE 로 승격). 두 상태의 차이·DB 확인·시나리오는 도메인 상태 운영 매뉴얼 참조.

스캔 전체 리포트(DiscoveryReport JSON). 조건부 GET 대상 — ETag 응답 헤더에 version 이 실린다(§3). 아직 완료된 스캔이 없으면 204 No Content. 200 응답에는 판단 근거가 각 finding 에 인라인된다 — 매칭되는 findings[i] 마다 classification·basis(§2.3 의 /discovery rationale 와 동일 shape)가 가산되고, reason 은 classification 뒤로 재배치된다(별도 top-level rationale[] 배열은 없다. basis shape 는 §2.3 /discovery/detail rationale 와 동일). 기존 report_json 필드 값은 불변이라 중앙 조건부 GET 은 깨지지 않는다.

curl -i http://<host>:8080/api/v1/domains/api.weble.net/scan-result/detail

2.4 스펙 · API 인벤토리 — /api/v1/domains/{host}/spec · /spec/apis

문서화 API 인벤토리는 스펙 리소스 아래에 둔다 — GET /spec=업로드된 문서 목록, GET /spec/apis=그 문서들에서 파싱한 API. 관측 트래픽 기반 발견(discovery) 결과(/scan-result/detail·/discovery)와는 구분된다.

PUT/api/v1/domains/{host}/spec  ·  GET/api/v1/domains/{host}/spec

PUT 은 스펙 원문(OpenAPI / Postman / CSV 바이트)을 Content-Type: application/octet-stream 본문으로 업로드한다. ?filename 은 곧 문서 식별자(specName)로 도출된다 — 정규화(trim·소문자), 미전달/빈 = "default". 서로 다른 filename = 별개 문서, 같은 filename 재업로드 = 그 문서의 새 버전이다.

★업로드 시 파싱된 API 목록·파라미터가 영속 API 인벤토리(documented_api)에 specName 단위로 reconcile 된다 — 이미 있으면 파라미터·deprecated·version 변경 시 갱신(UPDATED), 신규면 추가(ADDED), 그 문서에서 빠지면 삭제 표시(status=DELETED, 행 보존). ★다른 filename 문서는 건드리지 않는다(union). 인벤토리 조회는 아래 GET /spec/apis.

GET 은 활성 스펙 문서 목록을 반환하며, 스펙이 없으면 404 가 아니라 빈 목록 { "count":0, "items":[] }(200)이다. 각 문서는 specName·filename·uploadedAt 등을 포함한다.

3. 조건부 GET (ETag) — 304 실증

중앙 서버는 변경이 없을 때 큰 리포트 본문을 다시 받지 않도록 조건부 GET 을 쓴다. /scan-result 의 version 을 If-None-Match 헤더에 담아 /scan-result/detail 에 보내면, 결과가 그대로면 304 Not Modified(본문 없음)가 온다.

# 1) version 확인
curl -s http://<host>:8080/api/v1/domains/api.weble.net/scan-result | grep version
#   "version":"829bfa4c628a7ede"

# 2) 그 version 을 If-None-Match 로 detail 에 — 변경 없으면 304
curl -i -H 'If-None-Match: "829bfa4c628a7ede"' \
  http://<host>:8080/api/v1/domains/api.weble.net/scan-result/detail
#   HTTP/1.1 304 Not Modified   ← 실측 확인됨 (본문 재전송 회피)

ETag 값은 따옴표를 포함한다("829bfa4c628a7ede"). version 이 달라지면(=새 스캔으로 결과 갱신) 200 OK + 새 본문 + 새 ETag 가 온다. /scan-result/detail·/scan-result 의 version 은 동일 ETag 다.
