1. 개요 · 엔드포인트 한눈에 보기

모든 도메인 API 는 http://<host>:8080/api/v1 아래에 있고, 헬스체크 등 운영 엔드포인트는 /actuator 아래에 있다. 응답은 JSON 이다. 아래 표의 ⚠️ Loki 표시가 있는 두 엔드포인트만 운영 Loki 를 호출(부하 발생)하며 나머지는 전부 DB 조회다.

2. 엔드포인트 상세 (사용법 · 실제 응답)

2.1 도메인 인벤토리 — /api/v1/domains

GET/api/v1/domains

페이지네이션 도메인 목록. 응답 body 는 JSON 배열(각 원소는 경량 DomainView)이고, 페이지 정보는 응답 헤더에 실린다. 파라미터 ?page(0-based, 기본 0)·?size(기본 1000, 범위 [1,1000]). host 오름차순 정렬(결정적).

curl -i 'http://<host>:8080/api/v1/domains?page=0&size=2'
# 응답 헤더
X-Total-Count: 45592
X-Total-Pages: 22796
X-Current-Page: 0

# 응답 body (배열, size=2)
[
  { "host":"0.0.0.0", "enabled":true, "hostnames":["AAI43","ALDI1", "..."],
    "intervalOverride":null, "specMergeStrategy":"MERGE", "basePathStrip":null, "spec":null },
  { "host":"...", "enabled":true, "hostnames":["..."], "spec":null }
]

전체 도메인이 4만+ 건이라 페이지네이션이 필수다(과거 무페이지 전건 반환은 폐지). X-Total-Pages·X-Current-Page 로 순회한다. 목록의 spec 은 경량 메타이며, 페이지당(최대 1000건) 스펙 조회가 있다(배치 로드는 TODO).

POST/api/v1/domains

도메인 등록. host 는 정규화(소문자·trim)되며 이미 있으면 409 Conflict. 성공 시 201 Created + 등록된 DomainView. (CLI -domain -register 로도 등록 가능.)

curl -X POST http://<host>:8080/api/v1/domains \
  -H 'Content-Type: application/json' \
  -d '{"host":"api.example.com","enabled":true}'

요청 body(DomainUpsert) 선택 필드: hostnames·intervalOverride·specMergeStrategy·basePathStrip.

GET/api/v1/domains/{host}

한 도메인의 상세(DomainDetailView). 목록(경량)과 달리 lastScanAt 과 도메인 현재 effective 설정(effectiveClassification)·활성 스펙 메타(spec)까지 보강해 준다.

curl http://<host>:8080/api/v1/domains/api.weble.net
{
  "host":"api.weble.net",
  "enabled":true,
  "hostnames":["AORV1","AORV2","PORV1","PORV2"],
  "intervalOverride":null,
  "specMergeStrategy":"MERGE",
  "basePathStrip":null,
  "spec":null,
  "lastScanAt":"2026-06-29T05:00:37.105715Z",
  "effectiveClassification":{
    "profile":"MIDDLE", "threshold":0.7, "weightsSource":"preset",
    "weights":{ "hostApiSubdomain":0.4, "corsPreflight":0.3, "..." :"… 14키" }
  }
}

스펙이 있으면 spec 은 {"format":"OPENAPI","specVersion":1,"endpointCount":2,"uploadedAt":"…","specName":"default","filename":"petstore-v1.json","active":true} 형태(없으면 null). effectiveClassification 읽는 법은 §2.3 참조.

PUT/api/v1/domains/{host}  ·  DELETE/api/v1/domains/{host}

PUT 은 부분 수정(PATCH 의미)이다 — body 에 전달한 필드만 반영하고 미전달 필드는 기존값을 유지한다. 전 항목을 다 보낼 필요가 없다. DELETE 는 삭제 후 204 No Content(없으면 404).

# 부분 수정 — enabled 만 비활성으로 (나머지 설정은 그대로 유지)
curl -X PUT http://<host>:8080/api/v1/domains/api.weble.net \
  -H 'Content-Type: application/json' \
  -d '{"enabled":false}'

# 삭제
curl -X DELETE http://<host>:8080/api/v1/domains/api.weble.net   # → 204

미전달 규칙: enabled 미전달=유지(명시 false 만 비활성), hostnames 미전달=유지·[]=비우기(명시 빈 배열), intervalOverride·specMergeStrategy·basePathStrip 미전달=유지. 응답은 수정된 DomainView.

2.2 스캔 상태 · 결과 · 즉시 스캔 — /scan-status · /result · /scan · /scan-now

GET/api/v1/domains/{host}/scan-status

가벼운 상태 메타. 중앙 서버가 version·lastScanAt 만 비교해 결과 재조회 여부를 결정한다. latestSpec 은 최근 업로드된 활성 스펙 요약(파일명·업로드 일시·파일에서 추출한 API 수)이며 스펙이 없으면 null. summary 카운트 의미는 §2.3·DB 스키마 매뉴얼 참조.

curl http://<host>:8080/api/v1/domains/api.weble.net/scan-status
{
  "host":"api.weble.net",
  "state":"idle",
  "lastScanAt":"2026-06-26T09:04:13.072008Z",
  "version":"1913dd805b705e7a",
  "summary":{"discovered":166,"active":0,"shadow":161,"zombie":0,"unused":0},
  "totalDropped":5,
  "latestSpec":null
}

latestSpec 보유 예: {"filename":"petstore-v1.json","uploadedAt":"…","endpointCount":2, …}. api.weble.net 은 스펙 미업로드라 null.

GET/api/v1/domains/{host}/result

스캔 전체 리포트(DiscoveryReport JSON). 조건부 GET 대상 — ETag 응답 헤더에 version 이 실린다(§3). 아직 완료된 스캔이 없으면 204 No Content. 200 응답에는 엔드포인트별 판단 근거 rationale 배열이 가산된다(§2.3 의 /discovery 와 동일 shape). 기존 report_json 필드는 불변이라 중앙 조건부 GET 은 깨지지 않는다.

curl -i http://<host>:8080/api/v1/domains/api.weble.net/result

응답 본문 상위 키와 핵심 부분(트림). rationale 가 끝에 가산된다.

{
  "host":"api.weble.net", "generatedAt":"…", "logWindow":{…}, "specVersion":…,
  "summary":{"discovered":166,"active":0,"shadow":161,"zombie":0,"unused":0,"lowConfidence":0},
  "findings":[
    { "host":"api.weble.net", "method":"GET", "pathTemplate":"/requests",
      "confidence":0.9, "reason":"트래픽 존재, 문서 내 매칭 템플릿 없음",
      "params":{ "query":[
          {"name":"campaignId","count":10075,"lenBuckets":["S"],"sensitive":false},
          {"name":"type","count":10075,"lenBuckets":["S"],"sensitive":false} ],
        "path":[] },
      "low_confidence":false }
  ],
  "droppedNonApi":{"excluded":0,"webForm":0,"lowScore":5,"total":5},
  "droppedByLimit":{…}, "droppedNonExistent":{…},
  "endpointKindSignal":{…}, "typeDistribution":{…}, "preflightSignal":{…}, "specSource":{…},
  "rationale":[ { "method":"…", "pathTemplate":"…", "classification":"…", "basis":{…} } ]
}

상위 키: host·generatedAt·logWindow·specVersion·summary·findings·droppedNonApi·droppedByLimit·droppedNonExistent·endpointKindSignal·typeDistribution·preflightSignal·specSource·rationale(가산).

★ 시점 차이 report_json 의 findings 는 스캔 시점 결과지만, 가산된 rationale 는 조회 시점 현재 설정으로 재계산된다 — 둘이 다를 수 있다. 또 304 캐시 body 의 rationale 는 갱신되지 않는다(ETag 는 report 만 추적). 의도된 트레이드오프다.
POST/api/v1/domains/{host}/scan ⚠️ Loki

온디맨드 재스캔 트리거(비동기). 운영 Loki 를 호출하므로 남발 금지. 즉시 202 Accepted 만 반환하고 결과는 따로 /scan-status·/result 로 조회한다(비동기 큐잉은 TODO). 즉시 결과가 필요하면 아래 /scan-now.

curl -i -X POST http://<host>:8080/api/v1/domains/api.weble.net/scan   # → 202 Accepted
POST/api/v1/domains/{host}/scan-now ⚠️ Loki

즉시 동기 스캔. Loki 를 호출해 스캔을 끝낸 뒤 결합 결과(findings + rationale + effectiveClassification)를 바로 반환한다(§2.3 /discovery 와 동일 shape). 미등록 도메인이면 자동 등록(enabled=true) 후 스캔한다. 파라미터 ?window(ISO-8601 Duration, 기본/상한 = scan.max-window).

curl -X POST 'http://<host>:8080/api/v1/domains/api.weble.net/scan-now?window=PT1H'
⚠️ 동기 Loki 호출 — busy 도메인 주의 응답이 올 때까지 스캔이 진행되므로 트래픽이 많은 도메인은 응답이 지연될 수 있다 — window 를 작게 잡는다. 부하보호(slice·throttle·동시성·백오프·예산)는 내부에서 준수되고 watermark 는 전진하지 않는다(온디맨드 스냅샷). 비동기로 던지고 싶으면 POST /scan(202). Loki 수집/분석 실패 시 502 Bad Gateway, host 누락 400.

2.3 결합 디스커버리(판단 근거) — /discovery

GET/api/v1/domains/{host}/discovery

검출 카탈로그와 스펙을 결합한 결과다. 일반 findings 외에 도메인의 현재 설정(effectiveClassification)과 엔드포인트별 판단 근거(rationale)가 함께 내려온다(신규 기능). 신호·가중치를 읽는 자세한 법은 API 판독 매뉴얼 §4.3 을 참조한다.

curl http://<host>:8080/api/v1/domains/api.weble.net/discovery

① 도메인 현재 설정 — effectiveClassification

"effectiveClassification":{
  "profile":"MIDDLE",
  "threshold":0.7,
  "weightsSource":"preset",
  "weights":{
    "hostApiSubdomain":0.4, "corsPreflight":0.3, "apiSegment":0.55, "graphqlSegment":0.55,
    "versionSegment":0.26, "pathIdSegment":0.15, "machineEndpoint":0.2, "writeMethod":0.34,
    "query":0.12, "nonBrowserUa":0.24, "staticAssetPenalty":-0.6, "repeatBonus":0.12,
    "pathHint":0.55, "responseTypeApi":0.25
  }
}

② 엔드포인트별 판단 근거 — rationale[] (findings 와 병렬)

rationale[i] 는 findings[i] 와 동일 순서·동일 identity(method·host·pathTemplate)다. 아래는 점수로 판정된 Shadow 의 실제 근거다(api.weble.net, findings·rationale 각 192건).

{
  "method":"POST", "host":"api.weble.net", "pathTemplate":"/qnas",
  "classification":"SHADOW",
  "basis":{
    "type":"score", "apiScore":1.0, "threshold":0.7, "gate":"ADMIT", "mode":"pathless",
    "signals":[
      {"key":"hostApiSubdomain","weight":0.4, "fired":true, "contribution":0.4},
      {"key":"corsPreflight",   "weight":0.3, "fired":true, "contribution":0.3},
      {"key":"writeMethod",     "weight":0.34,"fired":true, "contribution":0.34},
      {"key":"repeatBonus",     "weight":0.12,"fired":true, "contribution":0.12}
      /* … 평가된 신호 전부(13개) — 미발화 신호는 "fired":false,"contribution":0.0 */
    ]
  }
}
읽는 법 — 위 예시 발화 신호 기여 합 0.4 + 0.3 + 0.34 + 0.12 = 1.16 → [0,1] clamp = apiScore 1.0 ≥ threshold 0.7 → gate ADMIT(API 로 확정 → 스펙에 없으므로 Shadow). mode:"pathless" 는 내장 path-shape 모드(operator 힌트 미설정). contribution 을 전부 더하면 apiScore 가 된다.

③ 분류별 근거가 다르다 — basis.type

점수(score)는 스펙에 없는 Shadow 에만 쓰인다. 스펙에 매칭되는 Active·Zombie 는 점수가 아니라 스펙 매칭이 근거다.

분류basis.type주요 필드근거
SHADOWscoreapiScore·threshold·gate·mode·signals[]점수 게이트 통과(ADMIT)
ACTIVEspec_matchspecRef·deprecated:false스펙 매칭(점수 아님)
ZOMBIEspec_matchspecRef·deprecated:true 또는 estimated:true스펙 매칭(deprecated/버전추정)
UNUSEDspec_onlyspecRef스펙에 있고 무트래픽
WEBPAGEendpoint_kindendpointKind·kindConfidence$type/referer 로 비-API(현재 미산출)

근거 설계 doc/34 §2. 비-API 로 drop 된 엔드포인트는 findings·rationale 에 없다(집계 droppedNonApi 만).

2.4 스펙 — /api/v1/domains/{host}/spec

PUT/api/v1/domains/{host}/spec  ·  GET/api/v1/domains/{host}/spec

PUT 은 스펙 원문(OpenAPI / Postman / CSV 바이트)을 업로드한다. ?filename(옵션)으로 원본 파일명을 함께 남길 수 있다(미전달 null). GET 은 활성 스펙 문서 목록(배열)을 반환하며, 스펙이 없으면 404 가 아니라 빈 배열 [](200)이다. 문서별 filename·uploadedAt 을 포함한다.

# 업로드(예: OpenAPI JSON, 파일명 기록)
curl -X PUT 'http://<host>:8080/api/v1/domains/api.example.com/spec?filename=petstore-v1.json' \
  -H 'Content-Type: application/octet-stream' \
  --data-binary @petstore-v1.json
# → {"format":"OPENAPI","specVersion":1,"endpointCount":2,"uploadedAt":"…","specName":"default","filename":"petstore-v1.json","active":true}

# 조회 — 스펙 미업로드 도메인(api.weble.net)
curl http://<host>:8080/api/v1/domains/api.weble.net/spec   # → [] (200)

# 조회 — 스펙 보유 도메인
curl http://<host>:8080/api/v1/domains/api.example.com/spec
[
  { "format":"OPENAPI", "specVersion":1, "endpointCount":2,
    "uploadedAt":"2026-06-29T07:35:40.406366Z", "specName":"default",
    "filename":"petstore-v1.json", "active":true }
]
★ Breaking 변경 이전 버전의 GET /spec 은 단일 객체 또는 404 였으나, 이제 배열(무스펙=[]·200)이다. 기존 소비자는 배열 처리로 바꿔야 한다.

api.weble.net 은 스펙이 없어([]) 모든 검출이 Shadow 로 분류된다(매칭 대조 대상 부재).

2.5 분류 설정 · 가중치 편집 — /classification

GET/api/v1/classification  ·  PUT/api/v1/classification

전역 분류 기본값(profile·임계·가중치·힌트). PUT 으로 수정한다. 도메인이 따로 override 하지 않으면 이 값을 상속한다.

curl http://<host>:8080/api/v1/classification
{
  "profile":"MIDDLE",
  "thresholdOverride":null,
  "customWeights":null,
  "matcher":null,
  "updatedAt":"2026-06-25T14:09:40.440546Z"
}

설정 필드 의미 — thresholdOverride · customWeights · matcher

필드의미적용 방식
thresholdOverrideAPI 점수 게이트 임계값([0,1]) 교체. Shadow(미문서) 후보가 이 값 이상이면 API 로 ADMIT.미설정(null)=preset 기본(MIDDLE 0.70 · HIGH 0.85 · LOW 0.55) 상속. 전 프로파일에서 설정 가능.
customWeights14개 신호 가중치 override map. 설정된 키만 교체·나머지는 베이스. 점수 = 발화 신호 가중치 합(clamp[0,1]).profile=CUSTOM 한정. unknown 키·비유한 값은 400 거부(WEIGHT_KEYS 14개만 허용).
matcher매칭/힌트 — api 힌트(경로 직접 선언 시 임계 무관 강제 ADMIT)·exclude(게이트 최우선 제외)·includeWebForms(write-method web-form 억제 on/off).explicit-hint 모드면 내장 path-shape 신호를 끄고 선언 경로만 pathHint 로 가산.

적용 우선순위: 도메인 override → 전역 → preset/MIDDLE(EffectiveClassificationResolver). 신호·가중치·모드 상세는 API 판독 매뉴얼 §4.3·§4.4.

GET/api/v1/domains/{host}/classification  ·  PUT/api/v1/domains/{host}/classification

도메인별 override. 응답은 override(그 도메인이 명시한 값, 없으면 전부 null)와 effective(상속까지 합산된 실효 설정)를 함께 준다. PUT 으로 도메인 override 를 설정한다.

curl http://<host>:8080/api/v1/domains/api.weble.net/classification
{
  "host":"api.weble.net",
  "override":{ "profile":null, "thresholdOverride":null, "customWeights":null, "matcher":null },
  "effective":{
    "profile":"MIDDLE",
    "weights":{ "hostApiSubdomain":0.4, "corsPreflight":0.3, /* …14키 */ "repeatMinCount":3 }
  }
}

override 가 전부 null = 전역(없으면 MIDDLE) 상속. 병합 우선순위(도메인 → 전역 → preset)는 API 판독 매뉴얼 §4.4 참조.

PATCH/api/v1/classification/weights  ·  PATCH/api/v1/domains/{host}/classification/weights

신호 가중치를 부분 편집한다. body = 편집할 키만 담은 weight map. ★편집하면 profile 이 자동으로 CUSTOM 이 되고, 편집하지 않은 키는 현재 effective 값을 유지한다(MIDDLE 로 리셋되지 않음). threshold·matcher 는 건드리지 않는다(그건 PUT /classification). unknown 키·비유한 값은 400, 도메인 PATCH 는 미등록 도메인이면 404.

# 전역 MIDDLE 도메인에 apiSegment 가중치만 0.7 로
curl -X PATCH http://<host>:8080/api/v1/classification/weights \
  -H 'Content-Type: application/json' \
  -d '{"apiSegment":0.7}'
{
  "profile":"CUSTOM",
  "thresholdOverride":null,
  "customWeights":{ "hostApiSubdomain":0.4, "corsPreflight":0.3, "apiSegment":0.7, "..." :"… 나머지 14키는 기존 effective 유지" },
  "matcher":null,
  "updatedAt":"…"
}
# effective.profile=CUSTOM, effective.weights.apiSegment=0.7 (편집 안 한 키는 그대로)

도메인 가중치는 PATCH /api/v1/domains/{host}/classification/weights 로 동일하게 편집한다(응답에 override·effective 동봉).

3. 조건부 GET (ETag) — 304 실증

중앙 서버는 변경이 없을 때 큰 리포트 본문을 다시 받지 않도록 조건부 GET 을 쓴다. /scan-status 의 version 을 If-None-Match 헤더에 담아 /result 에 보내면, 결과가 그대로면 304 Not Modified(본문 없음)가 온다.

# 1) version 확인
curl -s http://<host>:8080/api/v1/domains/api.weble.net/scan-status | grep version
#   "version":"1913dd805b705e7a"

# 2) 그 version 을 If-None-Match 로 result 에 — 변경 없으면 304
curl -i -H 'If-None-Match: "1913dd805b705e7a"' \
  http://<host>:8080/api/v1/domains/api.weble.net/result
#   HTTP/1.1 304 Not Modified   ← 실측 확인됨 (본문 재전송 회피)

ETag 값은 따옴표를 포함한다("1913dd805b705e7a"). version 이 달라지면(=새 스캔으로 결과 갱신) 200 OK + 새 본문 + 새 ETag 가 온다. /result·/scan-status 의 version 은 동일 ETag 다.
