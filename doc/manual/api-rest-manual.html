1. 개요 · 엔드포인트 한눈에 보기

2. 엔드포인트 상세 (사용법 · 실제 응답)

2.1 도메인 인벤토리 — /api/v1/domains

GET/api/v1/domains

페이지네이션 도메인 목록. 응답 body 는 JSON 배열(각 원소는 경량 DomainView)이고, 페이지 정보는 응답 헤더에 실린다. 파라미터 ?page(0-based, 기본 0)·?size(기본 1000, 범위 [1,1000]). host 오름차순 정렬(결정적).

curl -i 'http://<host>:8080/api/v1/domains?page=0&size=2'
# 응답 헤더
X-Total-Count: 51353
X-Total-Pages: 25677
X-Current-Page: 0

# 응답 body (배열, size=2)
[
  { "host":"0.0.0.0", "enabled":true, "hostnames":["AAI43","ALDI1","ALI41","AOC3-ICN-2501M0"],
    "intervalOverride":null, "specMergeStrategy":"MERGE", "basePathStrip":null, "spec":null },
  { "host":"00.674e7e49.v3-202405.cbricdns.com", "enabled":true, "hostnames":["AYJ11"],
    "intervalOverride":null, "specMergeStrategy":"MERGE", "basePathStrip":null, "spec":null }
]

전체 도메인이 4만+ 건이라 페이지네이션이 필수다(과거 무페이지 전건 반환은 폐지). X-Total-Pages·X-Current-Page 로 순회한다. 목록의 spec 은 경량 메타이며, 페이지당(최대 1000건) 스펙 조회가 있다(배치 로드는 TODO).

POST/api/v1/domains

도메인 등록. host 는 정규화(소문자·trim)되며 이미 있으면 409 Conflict. 성공 시 201 Created + 등록된 DomainView. (CLI -domain -register 로도 등록 가능.)

curl -X POST http://<host>:8080/api/v1/domains \
  -H 'Content-Type: application/json' \
  -d '{"host":"api.example.com","enabled":true}'

요청 body(DomainUpsert) 선택 필드: hostnames·intervalOverride·specMergeStrategy·basePathStrip.

GET/api/v1/domains/{host}

한 도메인의 상세(DomainDetailView). 목록(경량)과 달리 lastScanAt 과 도메인 현재 effective 설정(effectiveClassification)·활성 스펙 메타(spec)까지 보강해 준다.

curl http://<host>:8080/api/v1/domains/api.weble.net
{
  "host":"api.weble.net",
  "enabled":true,
  "hostnames":["AORV1","AORV2","PORV1","PORV2"],
  "intervalOverride":null,
  "specMergeStrategy":"MERGE",
  "basePathStrip":null,
  "spec":null,
  "lastScanAt":"2026-06-29T05:00:37.105715Z",
  "effectiveClassification":{
    "profile":"MIDDLE", "threshold":0.7, "weightsSource":"preset",
    "weights":{ "hostApiSubdomain":0.4, "corsPreflight":0.3, "apiSegment":0.55, "graphqlSegment":0.55,
                "versionSegment":0.26, "pathIdSegment":0.15, "machineEndpoint":0.2, "writeMethod":0.34,
                "query":0.12, "nonBrowserUa":0.24, "staticAssetPenalty":-0.6, "repeatBonus":0.12,
                "pathHint":0.55, "responseTypeApi":0.25 }
  }
}

스펙이 있으면 spec 은 {"format":"OPENAPI","specVersion":1,"endpointCount":2,"uploadedAt":"2026-06-29T07:35:40.406366Z","specName":"petstore-v1.json","filename":"petstore-v1.json","active":true} 형태(없으면 null). effectiveClassification 읽는 법은 §2.3 참조.

PUT/api/v1/domains/{host}  ·  DELETE/api/v1/domains/{host}

PUT 은 부분 수정(PATCH 의미)이다 — body 에 전달한 필드만 반영하고 미전달 필드는 기존값을 유지한다. 전 항목을 다 보낼 필요가 없다. DELETE 는 삭제 후 204 No Content(없으면 404).

# 부분 수정 — enabled 만 비활성으로 (나머지 설정은 그대로 유지)
curl -X PUT http://<host>:8080/api/v1/domains/api.weble.net \
  -H 'Content-Type: application/json' \
  -d '{"enabled":false}'

# 삭제
curl -X DELETE http://<host>:8080/api/v1/domains/api.weble.net   # → 204

미전달 규칙: enabled 미전달=유지(명시 false 만 비활성), hostnames 미전달=유지·[]=비우기(명시 빈 배열), intervalOverride·specMergeStrategy·basePathStrip 미전달=유지. 응답은 수정된 DomainView.

★ enabled ≠ 스캔 (D82)enabled 는 activity_status(무접속 시 자동 강등되는 활동 게이트)와 별개 축이다. 주기 스캔 대상 = enabled=true AND activity_status='ACTIVE' — enabled 를 켜도 무접속으로 INACTIVE 상태면 스캔되지 않는다. 즉시 스캔·주기 재포함은 §2.2 /scan-now 또는 트래픽 재개(다음 discovery 가 ACTIVE 로 승격). 두 상태의 차이·DB 확인·시나리오는 도메인 상태 운영 매뉴얼 참조.

2.2 스캔 상태 · 결과 · 즉시 스캔 — /scan-status · /result · /scan · /scan-now

GET/api/v1/domains/{host}/scan-status

가벼운 상태 메타. 중앙 서버가 version·lastScanAt 만 비교해 결과 재조회 여부를 결정한다. latestSpec 은 최근 업로드된 활성 스펙 요약(파일명·업로드 일시·파일에서 추출한 API 수)이며 스펙이 없으면 null. summary 카운트 의미는 §2.3·DB 스키마 매뉴얼 참조.

curl http://<host>:8080/api/v1/domains/api.weble.net/scan-status
{
  "host":"api.weble.net",
  "state":"idle",
  "lastScanAt":"2026-06-29T05:00:37.105715Z",
  "version":"829bfa4c628a7ede",
  "summary":{"discovered":162,"active":0,"shadow":153,"zombie":0,"unused":0},
  "totalDropped":9,
  "latestSpec":null
}

latestSpec 보유 예: {"filename":"petstore-v1.json","uploadedAt":"2026-06-29T07:35:40.406366Z","endpointCount":2}. api.weble.net 은 스펙 미업로드라 null.

GET/api/v1/domains/{host}/result

스캔 전체 리포트(DiscoveryReport JSON). 조건부 GET 대상 — ETag 응답 헤더에 version 이 실린다(§3). 아직 완료된 스캔이 없으면 204 No Content. 200 응답에는 판단 근거가 각 finding 에 인라인된다 — 매칭되는 findings[i] 마다 classification·basis(§2.3 의 /discovery rationale 와 동일 shape)가 가산된다(별도 top-level rationale[] 배열은 없다). 기존 report_json 필드는 불변이라 중앙 조건부 GET 은 깨지지 않는다.

curl -i http://<host>:8080/api/v1/domains/api.weble.net/result

응답 본문 상위 키와 핵심 부분(트림). 각 finding 에 classification·basis 가 가산된다.

{
  "host":"api.weble.net",
  "generatedAt":"2026-06-29T05:00:37.105714556Z",
  "logWindow":{"from":"2026-06-19T07:48:48.580072Z","to":"2026-06-19T08:18:48.580072Z"},
  "specVersion":0,
  "summary":{"discovered":162,"active":0,"shadow":153,"zombie":0,"unused":0,"lowConfidence":0},

      "confidence":0.9, "reason":"트래픽 존재, 문서 내 매칭 템플릿 없음",
      "params":{ "query":[], "path":[] },
      "low_confidence":false,
      "classification":"SHADOW",
      "basis":{ /* §2.3 ②의 basis 와 동일 구조: type·mode·gate·apiScore·threshold·signals(전체) */ } }
    /* 총 153건(Shadow) — 매칭되는 각 건에 classification·basis 가산 */
  ],
  "droppedNonApi":{"excluded":0,"webForm":0,"lowScore":9,"total":9},
  "droppedByLimit":{"templates":0,"params":0,"total":0},
  "droppedNonExistent":{"notFound":0},
  "endpointKindSignal":{"status":"DORMANT","staticRatio":0.0,"refererPresentRatio":0.994},
  "typeDistribution":{"top":[{"type":"document","count":346501}],"other":0},
  "preflightSignal":{"status":"DORMANT","acrmPresentOptions":0},
  "specSource":{"specVersion":0,"format":null,"warnings":[],"documents":[]}

상위 키: host·generatedAt·logWindow·specVersion·summary·findings·droppedNonApi·droppedByLimit·droppedNonExistent·endpointKindSignal·typeDistribution·preflightSignal·specSource. 판단 근거는 각 finding 에 인라인(classification·basis)되며 별도 top-level 배열은 없다 — 현재 설정과 identity(method·host·pathTemplate)가 매칭되는 finding 에만 가산된다.

★ 시점 차이 report_json 의 findings 는 스캔 시점 결과지만, 가산된 basis 는 조회 시점 현재 설정으로 재계산된다 — 둘이 다를 수 있다. 또 304 캐시 body 의 basis 는 갱신되지 않는다(ETag 는 report 만 추적). 의도된 트레이드오프다.
POST/api/v1/domains/{host}/scan ⚠️ Loki

온디맨드 재스캔 트리거(비동기). 운영 Loki 를 호출하므로 남발 금지. 즉시 202 Accepted 만 반환하고 결과는 따로 /scan-status·/result 로 조회한다(비동기 큐잉은 TODO). 즉시 결과가 필요하면 아래 /scan-now.

curl -i -X POST http://<host>:8080/api/v1/domains/api.weble.net/scan   # → 202 Accepted
POST/api/v1/domains/{host}/scan-now ⚠️ Loki

즉시 동기 스캔. Loki 를 호출해 스캔을 끝낸 뒤 결합 결과(findings + rationale + effectiveClassification)를 바로 반환한다(§2.3 /discovery 와 동일 shape). 미등록 도메인이면 자동 등록(enabled=true) 후 스캔한다. 파라미터 ?window(ISO-8601 Duration, 기본/상한 = scan.max-window).

curl -X POST 'http://<host>:8080/api/v1/domains/api.weble.net/scan-now?window=PT1H'
⚠️ 동기 Loki 호출 — busy 도메인 주의 응답이 올 때까지 스캔이 진행되므로 트래픽이 많은 도메인은 응답이 지연될 수 있다 — window 를 작게 잡는다. 부하보호(slice·throttle·동시성·백오프·예산)는 내부에서 준수되고 watermark 는 전진하지 않는다(온디맨드 스냅샷). 비동기로 던지고 싶으면 POST /scan(202). Loki 수집/분석 실패 시 502 Bad Gateway, host 누락 400.

2.3 결합 디스커버리(판단 근거) — /discovery

GET/api/v1/domains/{host}/discovery

검출 카탈로그와 스펙을 결합한 결과다. 일반 findings 외에 도메인의 현재 설정(effectiveClassification)과 엔드포인트별 판단 근거(rationale)가 함께 내려온다(신규 기능). 신호·가중치를 읽는 자세한 법은 API 판독 매뉴얼 §4.3 을 참조한다.

curl http://<host>:8080/api/v1/domains/api.weble.net/discovery

① 도메인 현재 설정 — effectiveClassification

"effectiveClassification":{
  "profile":"MIDDLE",
  "threshold":0.7,
  "weightsSource":"preset",
  "weights":{
    "hostApiSubdomain":0.4, "corsPreflight":0.3, "apiSegment":0.55, "graphqlSegment":0.55,
    "versionSegment":0.26, "pathIdSegment":0.15, "machineEndpoint":0.2, "writeMethod":0.34,

② 엔드포인트별 판단 근거 — rationale[] (findings 와 병렬)

rationale[i] 는 findings[i] 와 동일 순서·동일 identity(method·host·pathTemplate)다. 아래는 점수로 판정된 Shadow 의 실제 근거다(api.weble.net, findings·rationale 각 194건). pathless 모드라 평가 신호는 13개다(pathHint 는 explicit-hint 모드 전용=14번째 가중치, pathless 에선 미평가). 미작동 신호도 모두 포함된다(fired:false·contribution:0.0).

{
  "method":"POST", "host":"api.weble.net", "pathTemplate":"/clips",
  "classification":"SHADOW",
  "basis":{
    "type":"score", "mode":"pathless", "gate":"ADMIT", "apiScore":1.0, "threshold":0.7,
    "signals":[
      {"key":"hostApiSubdomain", "fired":true,  "weight":0.4,  "contribution":0.4},
      {"key":"corsPreflight",    "fired":true,  "weight":0.3,  "contribution":0.3},
      {"key":"apiSegment",       "fired":false, "weight":0.55, "contribution":0.0},
      {"key":"graphqlSegment",   "fired":false, "weight":0.55, "contribution":0.0},
      {"key":"versionSegment",   "fired":false, "weight":0.26, "contribution":0.0},
      {"key":"pathIdSegment",    "fired":false, "weight":0.15, "contribution":0.0},
      {"key":"machineEndpoint",  "fired":false, "weight":0.2,  "contribution":0.0},
      {"key":"writeMethod",      "fired":true,  "weight":0.34, "contribution":0.34},
      {"key":"query",            "fired":false, "weight":0.12, "contribution":0.0},
      {"key":"nonBrowserUa",     "fired":false, "weight":0.24, "contribution":0.0},
      {"key":"repeatBonus",      "fired":false, "weight":0.12, "contribution":0.0},
      {"key":"staticAssetPenalty","fired":false,"weight":-0.6, "contribution":0.0},
      {"key":"responseTypeApi",  "fired":false, "weight":0.25, "contribution":0.0}

읽는 법 — 위 예시 작동 신호 기여 합 0.4 + 0.3 + 0.34 = 1.04 → [0,1] clamp = apiScore 1.0 ≥ threshold 0.7 → gate ADMIT(API 로 확정 → 스펙에 없으므로 Shadow). 위 예에서 작동한 신호는 hostApiSubdomain·corsPreflight·writeMethod 3개이고 나머지 10개는 미작동(contribution 0.0)다. mode:"pathless" 는 내장 path-shape 모드(operator 힌트 미설정). contribution 을 전부 더하면 apiScore 가 된다.

③ 분류별 근거가 다르다 — basis.type

점수(score)는 스펙에 없는 Shadow 에만 쓰인다. 스펙에 매칭되는 Active·Zombie 는 점수가 아니라 스펙 매칭이 근거다.

분류basis.type주요 필드근거

2.4 스펙 · API 인벤토리 — /api/v1/domains/{host}/spec · /spec/apis

문서화 API 인벤토리는 스펙 리소스 아래에 둔다 — GET /spec=업로드된 문서 목록, GET /spec/apis=그 문서들에서 파싱한 API. 관측 트래픽 기반 발견(discovery) 결과(/result·/discovery)와는 구분된다.

PUT/api/v1/domains/{host}/spec  ·  GET/api/v1/domains/{host}/spec

PUT 은 스펙 원문(OpenAPI / Postman / CSV 바이트)을 Content-Type: application/octet-stream 본문으로 업로드한다. ?filename 은 곧 문서 식별자(specName)로 도출된다 — 정규화(trim·소문자), 미전달/빈 = "default". 서로 다른 filename = 별개 문서, 같은 filename 재업로드 = 그 문서의 새 버전이다.

★업로드 시 파싱된 API 목록·파라미터가 영속 API 인벤토리(documented_api)에 specName 단위로 reconcile 된다 — 이미 있으면 파라미터·deprecated·version 변경 시 갱신(UPDATED), 신규면 추가(ADDED), 그 문서에서 빠지면 삭제 표시(status=DELETED, 행 보존). ★다른 filename 문서는 건드리지 않는다(union). 인벤토리 조회는 아래 GET /spec/apis.

GET 은 활성 스펙 문서 목록(배열)을 반환하며, 스펙이 없으면 404 가 아니라 빈 배열 [](200)이다. 각 문서는 specName·filename·uploadedAt 등을 포함한다.

★ breaking 판정 — lastChangeBreaking · changedParams (?breaking=true)

현재 규칙 목록. 응답은 extensions·nameTokens 두 배열이다(확장자는 leading . 로 저장된다).

curl http://<host>:8080/api/v1/config/static-classify
{
  "extensions":[".js",".css",".png",".jpg",".jpeg",".gif",".svg",".ico",".webp",".woff",".woff2",".map"],
  "nameTokens":["img","image","thumb","thumbnail","resize","icon","logo","banner","sprite","avatar","favicon","css","link","download","attachment"]
}
POST/api/v1/config/static-classify  ·  DELETE/api/v1/config/static-classify/{kind}/{value}

POST 는 규칙을 추가(멱등)하고 즉시 재적용한 뒤 201 Created + 갱신된 목록을 반환한다. body 는 {"kind":"EXTENSION"|"NAME_TOKEN", "value":"..."}. value 는 trim·소문자 정규화되고 EXTENSION 은 leading . 이 없으면 보정된다(woff3 → .woff3). kind 가 두 값 밖이거나 value 가 빈 값이면 400. DELETE 는 경로의 {kind}/{value} 규칙을 삭제하고 즉시 재적용한 뒤 204 No Content(없으면 404).

# 확장자 규칙 추가 (value 는 .woff3 로 정규화됨)
curl -X POST http://<host>:8080/api/v1/config/static-classify \
  -H 'Content-Type: application/json' \
  -d '{"kind":"EXTENSION","value":"woff3"}'   # → 201 + 갱신된 목록

# 규칙 삭제
curl -X DELETE http://<host>:8080/api/v1/config/static-classify/NAME_TOKEN/sprite   # → 204
POST/api/v1/config/static-classify/reload

DB 규칙을 classifier 에 재적용한다(외부에서 DB 를 직접 수정한 경우 반영용). 갱신된 목록(200)을 반환한다.

curl -X POST http://<host>:8080/api/v1/config/static-classify/reload

3. 조건부 GET (ETag) — 304 실증

중앙 서버는 변경이 없을 때 큰 리포트 본문을 다시 받지 않도록 조건부 GET 을 쓴다. /scan-status 의 version 을 If-None-Match 헤더에 담아 /result 에 보내면, 결과가 그대로면 304 Not Modified(본문 없음)가 온다.

# 1) version 확인
curl -s http://<host>:8080/api/v1/domains/api.weble.net/scan-status | grep version
#   "version":"829bfa4c628a7ede"

# 2) 그 version 을 If-None-Match 로 result 에 — 변경 없으면 304
curl -i -H 'If-None-Match: "829bfa4c628a7ede"' \
  http://<host>:8080/api/v1/domains/api.weble.net/result
#   HTTP/1.1 304 Not Modified   ← 실측 확인됨 (본문 재전송 회피)

ETag 값은 따옴표를 포함한다("829bfa4c628a7ede"). version 이 달라지면(=새 스캔으로 결과 갱신) 200 OK + 새 본문 + 새 ETag 가 온다. /result·/scan-status 의 version 은 동일 ETag 다.
