도메인 상태 운영 매뉴얼 — enabled · activity_status

도메인의 두 상태 축(enabled=사용자 수동 스위치, activity_status=시스템 자동 활동 게이트)의 의미·차이·DB 확인·변경 방법. 근거 doc/43-inactive-domain-status.md·DECISIONS D82. DB 접속·REST 는 §5·§6.

핵심 한 줄. 주기 스캔 대상 = enabled = true AND activity_status = 'ACTIVE' AND NOT ghost_suppressed. 세 조건을 모두 만족해야 스캔한다.
★ 세 번째 축 — ghost_suppressed (D83)2026-07-22 추가. 스캔을 반복해도 self-endpoint 가 0(봇/foreign-host/스푸핑 Host)인 도메인을 자동 억제하는 플래그다. enabled·activity_status 와 별개 축. discovery 틱 게이트가 set(discovered_at이 scan.ghost-after=기본 7일보다 오래 + 스캔이력 + 0-endpoint + 사용자설정/스펙 없음), 수동 스캔(scan-now/scan)이 해제(가역). ACTIVE 라도 ghost_suppressed=true 면 스캔 안 한다. 확인: SELECT ... ghost_suppressed FROM domain_config. 해제: POST /scan-now. 근거 DECISIONS D83·doc/43 §8.

1. 두 상태는 무엇이 다른가

enabled 와 activity_status 는 다른 주체가·다른 이유로 설정하는 별개 컬럼(축)이다. 하나가 다른 하나를 대체하지 않는다.

enabled (boolean)activity_status (ACTIVE / INACTIVE)
의미"이 도메인을 관리 대상으로 켤까" — 수동 on/off 스위치"지금 실요청이 들어오는가" — 활동 게이트
설정 주체사람 (REST API·수동)시스템 자동 (D82)
변경 계기운영자가 명시적으로 켜고 끔무접속 7일 → INACTIVE(sweep) / 실요청 재개·수동 스캔 → ACTIVE
자동 변경안 됨 — discovery·스캔이 절대 안 건드림(doc/30 §5 불변식)됨 — discovery 틱(10분)마다 자동 전이
기본값true (신규 등록 시)ACTIVE (신규·기존행 마이그레이션 시)

무접속 판정 임계는 scan.inactive-after(기본 P7D=7일, 설정 override 가능). 이 값보다 마지막 실요청(last_seen_at)이 오래되면 discovery 틱 종료 sweep 이 ACTIVE → INACTIVE 로 강등한다.

2. 네 가지 조합과 스캔 여부

enabledactivity_status주기 스캔의미 / 대표 사례
trueACTIVE스캔함켜짐 + 트래픽 있음 = 정상 스캔 대상
trueINACTIVE스캔 안 함켜져 있지만 7일 무요청 → 강등. 요청 재개/수동 스캔 시 자동 ACTIVE 복귀
falseINACTIVE스캔 안 함사람이 끔(또는 정리된 유령 잔존)
falseACTIVE스캔 안 함꺼져 있음 = 트래픽 있어도 스캔 안 함(enabled=false 우선)
가장 헷갈리는 지점. enabled=true 라도 트래픽이 없어 INACTIVE 면 스캔되지 않는다. 반대로 트래픽이 있어도 enabled=false 면 스캔되지 않는다. "켜 놨는데 왜 스캔 안 되지?" 의 답은 대개 activity_status=INACTIVE 다.

참고 실측(2026-07-20 배포 직후 스냅샷, 시간에 따라 변동) — true·ACTIVE ~20,238 / true·INACTIVE ~16,451 / false·INACTIVE ~20,880 / false·ACTIVE ~1.

3. 무엇이 무엇을 바꾸나 (전이 규칙)

재활성(INACTIVE → ACTIVE)은 전부 activity_status 축에서만 일어난다. enabled 는 사람만 바꾼다.

이벤트activity_statusenabled
수동 스캔 (REST /scan-now·/scan)→ ACTIVE (항상)기존 도메인이면 불변. 미등록 신규일 때만 등록하며 true
로그에 실요청 관측 (discovery 10분)→ ACTIVE기존 도메인이면 불변. 신규 Host 면 등록하며 true
7일 무요청 (sweep)→ INACTIVE불변
enabled 켜고 끄기불변사람이 REST PUT/POST 로만
오해 주의. "수동 스캔하면 enabled 가 자동으로 켜진다"는 사실이 아니다. 수동 스캔·트래픽이 올리는 것은 activity_status 뿐이다. enabled 는 사용자 소유 설정이라 시스템이 자동으로 켜거나 끄지 않는다(신규 도메인 최초 등록 때만 true 로 붙는다).

"실요청"에서 프로브(status 404·470)와 헬스성 경로(/.cloudbric/pron/·/.cloudbric/afc/)는 제외된다(doc/42 C·doc/43 §4.1). 이 경로만 들어오는 도메인은 트래픽으로 안 쳐서 7일 뒤 INACTIVE 로 내려간다.

4. DB 에서 확인하는 법

접속(로컬 podman 에서):

podman run --rm -it -e PGPASSWORD=adc postgres:16-alpine \
  psql -h 192.168.8.197 -U adc -d adc

컬럼·제약 구조

\d domain_config
-- enabled                     boolean
-- activity_status             varchar NOT NULL DEFAULT 'ACTIVE'   (CHECK: ACTIVE|INACTIVE)
-- activity_status_changed_at  timestamptz   -- 마지막 전이 시각(null=이력 없음)
-- last_seen_at                timestamptz   -- discovery 가 본 마지막 실요청 시각(sweep 입력)

상태별 개수 (2×2)

SELECT enabled, activity_status, count(*)
FROM domain_config GROUP BY 1,2 ORDER BY 1 DESC, 2;

실제 스캔 대상 개수

SELECT count(*) FROM domain_config
WHERE enabled AND activity_status = 'ACTIVE';

특정 도메인 상태 + 전이 시각 + 마지막 실요청

SELECT host, enabled, activity_status, activity_status_changed_at, last_seen_at
FROM domain_config WHERE host = '<도메인>';

REST 로도 확인 (단건)

8080 은 VM 내부 전용(외부 방화벽) → ssh 로 실행. 응답의 activityStatus·activityStatusChangedAt 필드.

ssh root@192.168.8.197 "curl -s localhost:8080/api/v1/domains/<도메인>"

5. enabled 상태 변경하기 (사람이)

enabled 는 REST PUT 이 정식 경로다. present-only 반영이라 body 에 넣은 필드만 바뀌고 나머지 설정(hostnames·intervalOverride 등)은 유지된다.

끄기 (disable)

ssh root@192.168.8.197 "curl -s -X PUT localhost:8080/api/v1/domains/<도메인> \
  -H 'Content-Type: application/json' -d '{\"enabled\":false}'"

켜기 (enable)

ssh root@192.168.8.197 "curl -s -X PUT localhost:8080/api/v1/domains/<도메인> \
  -H 'Content-Type: application/json' -d '{\"enabled\":true}'"
  • host 는 자동 정규화(소문자·trim)된다.
  • 미등록 도메인은 PUT 이 404 → 새로 만들려면 POST /api/v1/domains {"host":"...","enabled":true} (이미 있으면 409).

DB 직접 (대안)

enabled 는 discovery 가 안 건드리므로 앱 가동 중 단일 행 UPDATE 도 안전하다.

UPDATE domain_config SET enabled = false, updated_at = now() WHERE host = '<도메인>';
★ enable 만으로는 스캔이 바로 안 될 수 있다. 스캔 조건은 enabled=true AND activity_status='ACTIVE' 인데, 꺼져 있던 도메인은 대개 INACTIVE 이기도 하다. enabled=true 만 주면 activity_status 가 INACTIVE 인 동안은 주기 스캔에 안 들어온다. 즉시 스캔·재포함까지 원하면 아래 §6 을 따른다.

6. 자주 겪는 시나리오

상황조치
켜져 있는데 무접속으로 스캔에서 빠짐 (true·INACTIVE)트래픽이 다시 들어오면 다음 discovery(10분)가 자동 ACTIVE 복귀. 즉시 원하면 POST /scan-now (1회 스캔 + ACTIVE 승격).
사람이 끈 도메인을 다시 스캔하고 싶음 (false)먼저 PUT {"enabled":true} (필수). 그다음 트래픽 대기 또는 POST /scan-now 로 ACTIVE 승격. enabled=false 인 채로는 스캔/트래픽만으론 절대 재포함 안 됨.
특정 도메인 스캔을 잠깐 멈추고 싶음PUT {"enabled":false}. (activity_status 와 무관하게 스캔 제외.)
지금 딱 한 번만 결과가 필요함 (등록·스캔 상태 안 바꾸고)POST /scan-now 는 온디맨드 스냅샷(워터마크 미전진)이나 activity_status 는 ACTIVE 로 올린다. 등록 자체를 원치 않으면 CLI one-off 사용.
.cloudbric/pron|afc 만 들어오는 도메인이 자꾸 INACTIVE 됨정상이다. 이 두 경로는 실요청으로 안 쳐서(doc/43 §4.1) 다른 트래픽이 없으면 7일 뒤 INACTIVE. 계속 스캔하려면 enabled 와 무관하게 실요청이 있어야 한다.

7. 즉시 스캔 트리거 요약 (REST)

엔드포인트동작activity_status워터마크
POST /api/v1/domains/{host}/scan-now동기 즉시 스캔 + 결합 결과 반환→ ACTIVE 승격미전진(스냅샷)
POST /api/v1/domains/{host}/scan비동기 트리거(202)→ ACTIVE 승격스케줄러 경로로 전진

둘 다 ScanSelector 게이트를 우회해 즉시 스캔한다(enabled·activity_status 무관하게 1회 실행). 단 enabled=false 면 승격돼도 주기 스캔엔 다시 안 들어온다(§6).
