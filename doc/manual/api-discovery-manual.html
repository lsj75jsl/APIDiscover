1. 개요 — 무엇을, 왜 탐지하는가

APIDiscover 는 방화벽(WAAP) 앞단 nginx 의 access log 만으로 API 인벤토리를 만들고, 문서(OpenAPI/Postman/CSV 스펙)와 대조해 관리되지 않는 API 를 찾아낸다. 로그에는 요청/응답의 본문(body)·헤더가 없으므로, "어떤 경로가 호출됐는가" 라는 관측 사실에서 출발한다.

구분정의집합식
Shadow트래픽엔 있는데 문서엔 없는 API (그림자 API)D \ S
Zombie문서상 deprecated(또는 구버전)인데 트래픽이 계속 도는 APIS_deprecated ∩ D
Active문서에 있고(미deprecated) 트래픽도 정상인 APIS ∩ D
Unused문서엔 있으나 트래픽이 관측되지 않은 APIS \ D

D = 로그에서 관측된 엔드포인트 집합(Discovered), S = 스펙 문서의 엔드포인트 집합(Spec).

핵심 어려움 로그 한 줄에는 "GET /users/123" 같은 호출 사실만 있다. 이것이 진짜 API 인지, 단순 웹 페이지/정적 자원(.js/.css)인지, 혹은 존재하지도 않는 경로를 두드린 스캐너 노이즈인지를 본문 없이 구분해야 한다. 이 매뉴얼의 4장이 그 판단 기준을 다룬다.

// LokiClient — 429/502/503/504 에 지수 백오프, 그 외 즉시 실패
for (int attempt = 0; attempt < MAX_ATTEMPTS; attempt++) {   // MAX_ATTEMPTS = 4
    throttle();                                              // 쿼리 간 최소 간격
    HttpResponse<String> response = send(request);          // 세마포어 acquire/release
    int status = response.statusCode();

        backoff(attempt);                                   // base * 2^attempt

3.2 ② 로그 파싱 — 24필드를 ParsedRequest 로

소스: parse/LogLineParser.java, model/ParsedRequest.java

실로그는 ^|^ 구분자로 최대 24개 필드를 갖는다. 파서는 고정 인덱스에서 핵심 필드를 읽고, HTTP 메서드가 아니거나 숫자/시간 파싱이 실패하는 손상 라인은 폐기한다.

필드인덱스용도
method[5] 토큰GET/POST/… (HTTP 메서드가 아니면 라인 폐기)
request_uri[8]경로 + 쿼리스트링 (쿼리는 분리 보관)
status[9]200 / 404 / 500 … (실재성·Zombie 신호)
referer[13]endpoint_kind 보조 신호 (nullable)
user_agent[14]SDK/CLI 판정 (non_browser_ua)
host[15] / [16]대상 도메인
$type[19]endpoint_kind 핵심 신호 — document/library 등 (nullable)
request_id[23]dedup 1순위 키 — 32 hex (nullable)
 
// LogLineParser.parse() — 핵심
String[] f = DELIM_PATTERN.split(line, -1);              // "^|^" 분리
if (f.length < FIELD_COUNT) return Optional.empty();      // FIELD_COUNT = 20 (코어)
String method = firstToken(f[F_REQUEST]);                // [5]
if (method == null || !HTTP_METHODS.contains(method)) return Optional.empty();

// type/requestId 는 24필드 로그에만 존재 — 없으면 null (20필드 로그 호환)
String type      = (f.length > F_TYPE)        ? nullIfDash(f[F_TYPE])        : null;  // [19]

개인정보·고카디널리티 방지 쿼리 파라미터는 이름 + 값 길이 버킷만 남기고 값 자체는 버린다. 본문이 없는 로그 특성과 결합해, 스키마/PII 는 애초에 수집 대상이 아니다.

3.3 ③ 중복 제거(dedup) · 증분 수집(watermark)

소스: batch/DiscoveryJobService.java (dedupByRequestId, windowFor), domain/Watermark.java

dedup: 윈도우 겹침·재시도로 같은 요청이 중복 수집될 수 있어, request_id 가 있으면 처음 본 것만 통과시킨다. request_id 가 없는 로그(20필드)는 그대로 보존한다.

static List<ParsedRequest> dedupByRequestId(List<ParsedRequest> requests) {
    Set<String> seen = new HashSet<>();
    List<ParsedRequest> out = new ArrayList<>(requests.size());
    for (ParsedRequest r : requests) {
        String id = r.requestId();
        if (id == null || seen.add(id)) out.add(r);   // id 없으면 무조건 보존
    }
    return out;
}

dedup 직후 두 가지를 추가로 걸러낸 뒤 인벤토리로 넘긴다 — ① foreign-host 제외: LogQL 의 |= 도메인 필터는 substring 전치필터라 referer/URL 에 도메인이 든 다른 Host 라인도 섞이므로, 파싱 후 스캔 도메인(정규화 동치)만 남긴다. ② /.cloudbric/pron/·/.cloudbric/afc/ 경로 제외(D82 정합): discovery 가 실요청에서 제외하는 WAAP 내부 경로는 스캔 인벤토리(discovered_endpoint)에도 넣지 않는다(discovery.excluded-paths 재사용, 단일 진실원).

watermark 증분: 도메인별로 "마지막으로 성공 분석한 윈도우 끝(lastEnd)"을 저장하고, 다음 스캔은 그 지점부터 신규 구간만 조회한다. 첫 스캔이면 initial-backfill 만큼 거슬러 올라간다.

// 윈도우 산정 (순수 함수)
static Optional<LogWindow> windowFor(Instant now, Instant lastEnd, Duration ingestLag, Duration backfill) {
    Instant end   = now.minus(ingestLag);                          // 로그 적재 지연 보정
    Instant start = (lastEnd != null) ? lastEnd : end.minus(backfill);  // 이어받기 / 첫 backfill
    if (!start.isBefore(end)) return Optional.empty();             // 신규 구간 없음 → 스킵
    return Optional.of(new LogWindow(start, end));
}

private static String classify(String seg) {
    if (UUID.matcher(seg).matches())    return "{uuid}";
    if (NUMERIC.matcher(seg).matches()) return "{id}";

4. API 로 확정하는 판단 기준 (ApiScorer)

소스: classify/ApiScorer.java — 이 장이 매뉴얼의 핵심이다.

스펙에 매칭되지 않은 관측 경로가 "보고할 만한 API(Shadow)"인지, 아니면 웹페이지/정적/노이즈로 버릴지를 가중 점수(api_confidence) 로 결정한다. 설계 철학은 "로그에 실제로 있는 신호만 쓰는 린(lean) 모델" — 본문/헤더가 없으니 Content-Type·AJAX 같은 강신호는 애초에 못 쓴다.

4.1 점수 모델 개념과 수식

각 신호가 양수/음수 가중치를 가지며, 발화한 신호의 가중치를 단순 합산해 [0,1] 로 clamp 한다(소수 3자리 반올림).

api_confidence = clamp[0,1] ( Σ wi · 𝟙(signali 발화) )
판정: api_confidence ≥ threshold(프로파일) → API 로 확정(ADMIT)

점수 자체는 "후보성(API 일 가능성)"이고, Shadow/Zombie 의 "실재성(confidence)"과는 역할이 분리돼 있다. 즉 게이트를 통과한 뒤에야 매칭·분류가 일어난다.

"모드" 컬럼의 의미 — 공통 · strict · explicit

점수 신호는 도메인의 힌트 설정에 따라 두 모드 중 하나로 평가된다. operator 가 api 경로 힌트(apiPathPrefixes/apiPathRegexes)를 하나라도 설정했는지(isExplicitHintMode)가 모드를 가른다.

모드언제 평가되나해당 신호
공통항상(두 모드 공통)hostApiSubdomain · corsPreflight · writeMethod · query · nonBrowserUa · repeatBonus · staticAssetPenalty · responseTypeApi

  • strict 와 explicit 는 상호배타다 — 한 도메인은 둘 중 하나만 적용된다. 공통 신호는 모드와 무관하게 항상 합산된다.
  • strict(기본): operator 가 아무 힌트도 설정하지 않은 상태. 경로 모양 휴리스틱(/api·/graphql·v1·{id}·health 등)으로 API 후보를 추정한다.
  • explicit: operator 가 api 경로를 직접 선언하면 내장 path-shape 신호를 끄고(오탐·중복 방지) 선언된 경로만 pathHint(0.55)로 점수에 반영한다. "선언적 단언이 휴리스틱보다 우선"이라는 원칙이며, 선언된 경로는 게이트에서 임계와 무관하게 강제 admit 된다(§4.6 게이트 · §4.7 명시 힌트 매처).
적용 · 사용 방법
  • 설정 위치: 중앙 REST 분류설정의 matcher 필드 — 전역 PUT /api/v1/classification 또는 도메인 PUT /api/v1/domains/{host}/classification(§4.4). 전역 ∪ 도메인 힌트가 합쳐져 해당 host 의 모드를 결정한다.
  • 언제 explicit 를 쓰나: 내장 휴리스틱이 못 잡는 API — 예를 들어 api 서브도메인·/api prefix·CORS 가 모두 없는 동일출처 www JSON API. 이때 해당 경로를 api 힌트로 선언하면 확실히 API 로 잡힌다.
  • 기본 권장: 별도 설정이 없으면 strict(기본)로 충분하다. explicit 는 휴리스틱의 사각지대를 보완할 때만 도입한다.

③ ★ 분류별 근거가 다르다 — 점수는 Shadow 에만

핵심: 모든 분류가 점수로 판정되는 게 아니다. 점수 게이트는 스펙에 없는 엔드포인트(Shadow 후보)를 API 로 인정할지에만 쓰인다. 스펙에 매칭되는 Active·Zombie 는 점수와 무관하게 스펙 매칭이 근거이고, Unused 는 스펙에만 있고 트래픽이 없다. 그래서 basis.type 이 분류마다 다르다.

4.4 도메인별 분류설정 override

§4.3 의 preset 은 전역 기본값일 뿐이다. 운영자는 도메인마다 profile·threshold(임계값)·weights(가중치)·matcher(힌트)를 개별 override 할 수 있다. 미설정 도메인은 전역(전역도 없으면 MIDDLE)을 상속하므로, 아무 설정이 없어도 현행과 100% 동일하다(무회귀).

병합 우선순위

한 호스트의 effective(실효) 설정은 도메인 override → (없으면) 전역 → (없으면) preset/MIDDLE 순으로 합쳐진다(EffectiveClassificationResolver).

profile = 도메인 ?? 전역 ?? MIDDLE
threshold = 도메인 ?? 전역 ?? preset(프로파일) // 어떤 프로파일에서도 override
weights = (profile == CUSTOM) ? [ MIDDLE 베이스 + 전역 ∪ 도메인(키 충돌=도메인 승) ] : preset 가중치
설정override 가능 조건병합 규칙
profile (HIGH/MIDDLE/LOW/CUSTOM)항상도메인 → 전역 → MIDDLE
threshold (임계값)어떤 프로파일에서도도메인 → 전역 → preset
weights (가중치)profile=CUSTOM 한정MIDDLE 베이스 + 전역 ∪ 도메인(키 충돌 시 도메인 승). HIGH/MIDDLE/LOW 는 preset 가중치 사용(override 무시)
matcher (api/exclude 힌트)항상전역 ∪ 도메인 합집합

즉 가중치 미세조정(weights)은 profile=CUSTOM 일 때만 의미가 있고, HIGH/MIDDLE/LOW 를 고른 도메인은 preset 가중치를 그대로 쓰되 임계값(threshold)만은 따로 조정할 수 있다.

중앙 REST API

전역·도메인 설정을 중앙 서버가 REST 로 읽고 쓴다(ClassificationController, @RequestMapping("/api/v1")).

4.5 핵심 코드 — score()

발화한 신호의 가중치를 더하는 단순·투명한 구조다. explicit-hint 모드 분기를 제외하면 모든 신호가 한 메서드에서 가산된다.

public double score(DiscoveredEndpoint d, boolean corsPreflight, ApiHintMatcher hints) {
    double s = 0.0;

    // host/cors — 양 모드 공통 (access log 기반 최강 양성 신호)
    if (hasApiHost(d))   s += w.hostApiSubdomain();
    if (corsPreflight)   s += w.corsPreflight();

    if (hints.isExplicitHintMode()) {
        // operator 가 경로 힌트를 줬으면 내장 path-shape 비활성, 힌트 매치만 가산
        if (hints.apiHinted(d.pathTemplate())) s += w.pathHint();
    } else {
        // 내장 path-shape 신호 (기본)
        String[] segs = segments(d.pathTemplate());
        if (hasApiSegment(segs))        s += w.apiSegment();
        if (hasGraphqlSegment(segs))    s += w.graphqlSegment();
        if (anyMatch(segs, VERSION_SEG)) s += w.versionSegment();
        if (anyIn(segs, ID_TOKENS))     s += w.pathIdSegment();
        if (anyIn(segs, MACHINE))       s += w.machineEndpoint();
    }

    // method/query/ua/repeat/static/responseType — 공통
    if (isWriteMethod(d))                              s += w.writeMethod();
    if (d.hadQuery())                                  s += w.query();
    if (d.nonBrowserUa())                              s += w.nonBrowserUa();
    if (d.metrics() != null && d.metrics().hits() >= w.repeatMinCount()) s += w.repeatBonus();
    if (d.endpointKind() == EndpointKind.STATIC                                        // 음수 감점
            || EndpointKindClassifier.hasStaticResourceName(d.pathTemplate())) s += w.staticAssetPenalty();  // img.php 류 포함
    if (d.endpointKind() == EndpointKind.API_CANDIDATE) s += w.responseTypeApi();     // 양성-only

    return Math.max(0.0, Math.min(1.0, Math.round(s * 1000.0) / 1000.0));   // clamp [0,1]
}

POST api.example.com/v2/orders?ref=1 (MIDDLE, hits 12, 쿼리 있음):

= 1.24 → clamp = 1.000 ≥ 0.70 → ADMIT (Shadow 후보)

4.6 게이트(gate) — 임계 통과 판정

소스: ApiScorer.evaluate()

점수만으로 결정하지 않는다. operator 의 명시 선언(exclude/hint)이 점수보다 우선하고, 웹폼 억제가 그 다음, 마지막이 점수 게이트다. 순서가 곧 우선순위다.

public Gate evaluate(DiscoveredEndpoint d, boolean corsPreflight, ApiHintMatcher hints) {
    if (hints.excluded(d.pathTemplate()))   return Gate.DROP_EXCLUDED;   // 1. 최우선 hard-drop
    if (hints.apiHinted(d.pathTemplate()))  return Gate.ADMIT;           // 2. hard-admit(임계 우회)
    if (!hints.includeWebForms()                                        // 3. web-form 억제
            && d.endpointKind() == EndpointKind.WEB_PAGE
            && isWriteMethod(d)
            && !hasStrongApiSignal(d, corsPreflight))                    // host_api·cors 강신호면 면제
        return Gate.DROP_WEB_FORM;
    return score(d, corsPreflight, hints) >= w.threshold()              // 4. 점수 게이트
            ? Gate.ADMIT : Gate.DROP_LOW_SCORE;
}

DROP_EXCLUDEDoperator 가 제외 선언 → 비API, 보고 안 함(메트릭만)
DROP_WEB_FORM웹폼 POST 추정 → 비API
DROP_LOW_SCORE임계 미달 → 비API(non_api 메트릭에만 집계)

스펙에 매칭되는 경로는 권위가 있으므로 게이트를 거치지 않고 곧장 Active/Zombie 판정으로 간다. OPTIONS 는 CORS 신호로만 쓰고 보고에서 제외된다.

4.7 명시 힌트 매처 · 404-only 필터

소스: match/ApiHintMatcher.java

점수 모델이 못 구제하는 경우(예: api 서브도메인·/api prefix·CORS 모두 없는 동일출처 www JSON API)를 위해, operator 가 경로를 직접 선언할 수 있다.

  • api 힌트(apiPathPrefixes/apiPathRegexes) — 매치 시 임계와 무관하게 강제 API 확정(hard-admit).
  • exclude(excludePathPrefixes/excludePathRegexes) — 매치 시 강제 제외(게이트 내 최우선).
  • api 힌트가 하나라도 설정되면 explicit-hint 모드 → 내장 path-shape 신호 비활성, pathHint 만 사용.
  • ReDoS 방어: prefix 200개·regex 50개·패턴 200자·입력 4096자 상한 + 매칭당 50ms deadline.

7. 현재 프로그램의 한계점

요약 탐지의 정확도는 로그에 담긴 신호의 양에 근본적으로 묶여 있다. 본문/헤더가 없는 access log 의 한계와, 점수 가중치가 아직 소규모 보정에 그친다는 점이 가장 큰 제약이다.

7.1 근본 제약 — 로그에 본문·헤더가 없음

  • 요청/응답 스키마·파라미터 타입 추론 불가(body 없음). 쿼리 파라미터도 이름+길이버킷만.
  • 인증 방식 식별·응답 본문 PII 노출 탐지 불가.
  • 따라서 산출물은 "엔드포인트 인벤토리 + Shadow/Zombie 판정"으로 범위가 제한된다.

7.2 $type(API_CANDIDATE) 신호가 실데이터에서 거의 안 잡힘

  • endpoint_kind 의 API 양성 신호(responseTypeApi)는 $type ∈ {xhr,fetch,json,api,ajax} 에 의존하는데, 실 Loki 샘플에서 이 값들이 사실상 0건으로 관측됐다(현재 가중치는 보정 전 1차값).
  • $type 의 전체 taxonomy 를 넓게 샘플링해 API_TYPES 집합을 확정하는 작업이 미완(후속 과제).
  • 결과적으로 현재 API 판정은 host/경로형태/메서드/CORS/반복 신호에 주로 기댄다.

7.3 동일출처(www) JSON API 분리의 어려움

  • api 서브도메인·/api prefix·CORS preflight 가 모두 없는 동일출처 www JSON API 는 자동 점수만으로 웹페이지와 분리하기 어렵다.
  • 현재는 operator 가 명시 경로 힌트(ApiHintMatcher) 로 보완해야 한다 — 완전 자동이 아니다.

7.4 점수 가중치가 잠정값(소규모 보정)

  • preset 가중치는 API 호스트 1개 vs 웹 호스트 1개의 실데이터로만 보정됐다. 다양한 도메인 대상 대규모 실데이터 보정은 보류 상태(임의값 성격 잔존).
  • responseTypeApi·Zombie severity 1차값도 보정 전.

7.5 분류·정규화의 구조적 오차

  • preflight vs 진짜 OPTIONS 구분 불가 — 스펙에 OPTIONS operation 이 있으면 Unused 오판 가능(preflightAmbiguous 플래그로 저신뢰 표시만).
  • 휴리스틱/통계 클러스터링은 과병합 위험이 있다(서로 다른 엔드포인트를 한 {var} 로 합칠 수 있어 신뢰도 감산으로 완화).
  • host 당 template 상한 5000 초과분은 hits 낮은 순으로 버려진다(저빈도 엔드포인트 누락 가능, 메트릭으로 노출).
  • request_id 가 없는 로그(20필드)는 dedup 이 불가능해 중복이 그대로 집계될 수 있다.

7.6 운영·확장성

  • HA 다중 실행 미보장 — 현재 단일 @Scheduled 만 사용. 다중 인스턴스 배포 시 같은 도메인을 중복 스캔할 수 있다(ShedLock/Quartz 클러스터 도입은 후속).
  • Spring Batch JobRepository 실연결 미완(batch.job.enabled=false, 스케줄러만 트리거).
  • 중앙 서버 연동의 서비스 간 인증 미구현(SecurityConfig permitAll) — 외부 연동은 자체 기능 안정화 후 진행 예정.
  • 재방문 주기는 티어(활성/기본/비활성/dormant) 로 자동 결정된다(D48, nextScanDueAt collapse). 도메인별 intervalOverride 도 이 커서에 반영된다. 갱신됨 (구 매뉴얼의 "전역 간격만 적용" 기술은 폐기).

8. 한계 개선 — nginx access log 추가 항목 (소비 구현 완료·활성 대기)

7장 한계의 상당수는 "로그에 담긴 신호가 부족"하기 때문이다. nginx 는 log_format 에 다양한 내장 변수를 추가로 남길 수 있다. 아래 항목 중 판별력이 확인된 것들의 소비 로직은 이미 구현·머지되어 있고(PR #73 / 05c17aa, D79 — 파서·EndpointKindClassifier·ApiScorer), 지금은 DORMANT(휴면) 상태로 대기한다. 운영자가 nginx log_format 에 필드를 추가하고 해당 인덱스를 설정하면(§8.4) 판정기가 그 신호를 소비하기 시작한다.

무회귀 — 지금은 현행 판정 100% 동일 모든 신규 필드 인덱스의 기본값은 -1(미사용)이다. 즉 파서가 해당 필드를 읽지 않으므로 전 신규 신호가 부재(0 기여)이고, 점수·endpoint_kind 판정은 기존과 완전히 동일하다. 신규 코드가 배포돼 있어도 log_format 을 바꾸지 않는 한 아무 것도 달라지지 않는다(ACRM 과 동일한 "있으면 읽는" 안전 설계). 활성화(nginx log_format 변경 + 인덱스 설정)는 별도의 운영 단계다(§8.4).
최우선 단 한 항목 — 응답 Content-Type $sent_http_content_type(응답 Content-Type 헤더) 하나만 추가해도 한계 7.2($type 신호 부재) 와 7.3(동일출처 www JSON API 분리) 가 거의 해소된다. 응답이 application/json·application/xml·application/grpc 면 — api 서브도메인이나 CORS 가 없어도 — 결정적으로 API 이고, text/html 이면 웹페이지, image/*·text/css·application/javascript 면 정적이다. 현재 의존하는 nginx $type(document/library, 실데이터에서 API 값 거의 0건)을 실제 응답 타입으로 대체한다. 단, 응답 CT 는 오도될 수 있어(에러 html·WAF·3xx) 완화 가드와 함께 소비한다(§8.3).

8.4 nginx log_format 예시 (append 인덱스 24~30)

기존 ^|^ 구분자 포맷의 끝(인덱스 24부터)에 새 필드를 덧붙이면 기존 필드 위치(0~23)가 보존되어 수집·파서 호환성을 유지한 채 점진 도입할 수 있다. 아래 append 순서는 ParseProperties 인덱스와 정확히 정합한다 — 순서가 곧 인덱스이므로 임의로 바꾸지 말 것. $server_protocol·$upstream_addr 은 미채택(§8.1)이라 넣지 않는다.

# 1) Authorization 에서 '스킴'만 추출 — 토큰 값은 로그에 남기지 않는다(보안)
map $http_authorization $auth_scheme {
    default       "-";
    "~*^bearer"   "bearer";
    "~*^basic"    "basic";
    "~*^digest"   "digest";
    "~."          "other";
}

log_format api_discovery
    '... 기존 24개 필드 0~23 (client_ip ~ request_id) ...'
    '^|^$sent_http_content_type'               # [24] 응답 Content-Type      ← 최우선(endpoint_kind)
    '^|^$content_type'                         # [25] 요청 본문 CT           (예약·미소비 — 슬롯 정렬용)
    '^|^$http_accept'                          # [26] 요청 Accept            → acceptJson
    '^|^$http_x_requested_with'                # [27] XHR 여부               → xRequestedWith
    '^|^$http_access_control_request_method'   # [28] CORS preflight (ACRM)
    '^|^$http_origin'                          # [29] CORS Origin            → originHeader
    '^|^$auth_scheme';                         # [30] 인증 스킴(토큰 제외)   → authScheme

access_log /var/log/nginx/access.log api_discovery;

위 변수는 모두 nginx 기본/표준 모듈이 제공한다. $sent_http_name 은 임의 응답 헤더, $http_name 은 임의 요청 헤더를 가리키며 헤더명의 - 는 _ 로, 대문자는 소문자로 바뀐다(예: Access-Control-Request-Method → $http_access_control_request_method). $content_type(25)은 현재 소비하지 않지만, 빼면 뒤 필드 인덱스가 한 칸씩 밀리므로 슬롯 정렬을 위해 그대로 둔다(파서는 25 에 인덱스를 부여하지 않아 무시).

활성화 — 코드 배포 후 별도 운영 단계 위 log_format 을 적용해도, 판정기가 소비하려면 배포 설정(adc.yaml 또는 application.yml)의 apidiscover.parse.* 인덱스를 함께 세팅해야 한다. 미설정 = -1 = DORMANT(무회귀). 
apidiscover:
  parse:
    response-content-type-field-index: 24   # $sent_http_content_type (★요청측 $content_type[25] 아님)
    accept-field-index: 26                   # $http_accept → acceptJson
    x-requested-with-field-index: 27         # $http_x_requested_with → xRequestedWith
    acrm-field-index: 28                     # $http_access_control_request_method (ACRM)
    origin-field-index: 29                   # $http_origin → originHeader
    auth-scheme-field-index: 30              # $auth_scheme → authScheme
보안·프라이버시 주의
  • $http_authorization·$http_cookie 의 원문(토큰·세션 값)을 로깅하지 말 것 — 위 map 처럼 스킴/존재 여부만 남긴다.
  • $request_body 와 쿼리 파라미터 값 은 PII·자격증명이 섞일 수 있어 로깅 금지(프로젝트 범위 밖 = body·PII 비수집 원칙 유지).
  • 응답 Content-Type 같은 메타데이터만 추가하는 것이 가장 안전하면서 효과가 크다.

8.7 그래도 남는 한계 — 본문(body) 미수집

8장의 항목들은 모두 헤더·메타데이터다. 즉 "이 호출이 API 인가"라는 판정(분류)은 크게 개선하지만, 요청/응답 본문(body)을 로그에 남기지 않는다는 근본 전제는 그대로다. 따라서 항목을 모두 추가하더라도 아래는 여전히 불가능하다.

헤더를 추가해도 해결되지 않는 것 (body 부재)
  • API 내부 계약(스키마) 복원 불가 — Content-Type: application/json 으로 "JSON 이다"는 알지만, 어떤 필드·타입·필수여부인지는 본문이 없어 알 수 없다.
  • 단일 엔드포인트에 숨은 작업 구분 불가 — GraphQL(/graphql)·JSON-RPC 는 본문의 query/method 로 작업이 갈리는데, URL·헤더만으로는 한 엔드포인트로만 보인다.
  • 응답 본문 PII 노출 탐지 불가 — 응답 타입은 알아도 본문에 어떤 민감정보가 담겨 나가는지는 볼 수 없다(프로젝트 범위 밖, 의도된 경계).
  • 파라미터 값·페이로드 검증 불가 — 쿼리/본문 값은 수집하지 않으므로(이름·길이버킷만) 실제 값 기반 분류·검증은 못 한다.
  • 선언과 실제의 불일치 미검증 — $sent_http_content_type 은 서버가 선언한 타입일 뿐, 본문이 실제로 그 형식인지 확인할 수 없다.
  • 인증 "스킴 존재"만 — 결과는 모름 — $auth_scheme 로 토큰 인증 사용 여부는 알아도, 인증 성공/실패·권한 수준은 본문/세부 헤더가 없어 판단 불가.

정리하면, 로그 항목 추가는 "무엇이 API 인가"(탐지·분류)의 정확도를 높이지만, "그 API 의 내부 계약(스키마·작업·데이터)"까지 복원하지는 못한다. 후자는 본문이 필요하고, 본문 미수집은 PII·성능·프라이버시를 위한 의도된 설계 경계다. 스키마·작업 수준 정보가 필요하면 access log 가 아니라 스펙 문서 업로드(OpenAPI/Postman)로 보완하는 것이 본 시스템의 전제다.
